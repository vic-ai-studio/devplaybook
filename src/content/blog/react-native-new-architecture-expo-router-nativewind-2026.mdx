# React Native New Architecture 2026: Expo Router v3 + NativeWind

React Native has undergone its most significant transformation since its initial release. As of 2026, the **New Architecture is the default** for all new projects — no more opt-in flags, no more legacy bridge. If you are starting a new mobile app today, you are already on JSI, Fabric, and TurboModules. This guide walks you through the complete modern stack: Expo SDK 52+, Expo Router v3, NativeWind v4, Reanimated 3, and EAS deployments.

---

## What Changed and Why It Matters

For years, React Native used a **JavaScript Bridge** — an asynchronous serialization layer that passed JSON messages between JS and native code. Every interaction (layout, gesture, module call) had to cross this bridge, introducing latency and jank.

The New Architecture replaces this with three core primitives:

### 1. JavaScript Interface (JSI)

JSI is a lightweight C++ layer that lets JavaScript hold **direct references** to native objects. No serialization. No async queue. You call a native function the same way you call any JS function.

```typescript
// Old Architecture (Bridge) — fire and forget, async
NativeModules.MyModule.getValue((result) => {
  console.log(result);
});

// New Architecture (JSI) — synchronous, direct
import { NativeMyModule } from "./specs/NativeMyModule";

const value = NativeMyModule.getValue(); // synchronous!
console.log(value);
```

### 2. Fabric (New Renderer)

Fabric is the new rendering system for React Native. It brings React 18 Concurrent Mode features — Suspense, transitions, and automatic batching — to mobile. Layouts are calculated on the C++ Shadow Tree and can be committed synchronously on the main thread when needed.

```typescript
// React 18 Concurrent features now work natively
import { Suspense, startTransition } from "react";

function App() {
  const [tab, setTab] = useState("home");

  function handleTabPress(nextTab: string) {
    startTransition(() => {
      setTab(nextTab); // non-urgent, deferred
    });
  }

  return (
    <Suspense fallback={<LoadingScreen />}>
      <TabContent tab={tab} />
    </Suspense>
  );
}
```

### 3. TurboModules

TurboModules replace the old NativeModules registry. They are lazily loaded — a module is not instantiated until your JS code first accesses it. This dramatically improves startup time for apps with many native dependencies.

```typescript
// TurboModule spec definition (TypeScript-first)
import type { TurboModule } from "react-native/Libraries/TurboModule/RCTExport";
import { TurboModuleRegistry } from "react-native";

export interface Spec extends TurboModule {
  multiply(a: number, b: number): number;
  fetchUser(id: string): Promise<{ name: string; email: string }>;
}

export default TurboModuleRegistry.getEnforcing<Spec>("MyTurboModule");
```

## Performance Comparison

| Metric | Old Architecture | New Architecture | Improvement |
|--------|-----------------|-----------------|-------------|
| JS-to-Native call | ~1-2ms (async) | ~0.01ms (sync JSI) | ~100x faster |
| App startup (cold) | 800-1200ms | 400-600ms | ~50% faster |
| Animation dropped frames | 5-15% | less than 1% | ~10x fewer drops |
| Memory overhead | High (bridge serialization) | Low (shared C++ objects) | ~30% reduction |
| Large list scroll | 55-58fps | 59-60fps | Consistently smooth |
| Concurrent renders | Not supported | Full React 18 support | New capability |

## Expo Router v3: File-Based Routing

Expo Router v3 brings Next.js-style file-based routing to React Native. Every file in the `app/` directory becomes a route. This works on iOS, Android, and the Web from a single codebase.

### Directory Structure

```
app/
├── _layout.tsx          # Root layout (providers, fonts)
├── index.tsx            # "/" — Home screen
├── (tabs)/
│   ├── _layout.tsx      # Tab navigator layout
│   ├── index.tsx        # "/tabs" — First tab
│   ├── explore.tsx      # "/tabs/explore" — Second tab
│   └── profile.tsx      # "/tabs/profile" — Third tab
├── (auth)/
│   ├── _layout.tsx      # Auth stack layout
│   ├── login.tsx        # "/auth/login"
│   └── register.tsx     # "/auth/register"
├── post/
│   ├── [id].tsx         # "/post/123" — Dynamic route
│   └── [id]/
│       └── comments.tsx # "/post/123/comments" — Nested
└── +not-found.tsx       # 404 fallback
```

### Root Layout

```typescript
// app/_layout.tsx
import { Stack } from "expo-router";
import { StatusBar } from "expo-status-bar";
import { GestureHandlerRootView } from "react-native-gesture-handler";

export default function RootLayout() {
  return (
    <GestureHandlerRootView style={{ flex: 1 }}>
      <Stack>
        <Stack.Screen name="(tabs)" options={{ headerShown: false }} />
        <Stack.Screen name="(auth)" options={{ headerShown: false }} />
        <Stack.Screen
          name="post/[id]"
          options={{ title: "Post", headerBackTitle: "Back" }}
        />
        <Stack.Screen name="+not-found" />
      </Stack>
      <StatusBar style="auto" />
    </GestureHandlerRootView>
  );
}
```

## Typed Navigation with Link and useLocalSearchParams

Expo Router v3 ships with **typed routes** out of the box. Enable them in `tsconfig.json`:

```json
{
  "compilerOptions": {
    "strict": true,
    "plugins": [{ "name": "expo/typescript-plugin" }]
  }
}
```

Now `Link` and `router.push` are fully typed:

```typescript
// Navigating to a typed route
import { Link, router } from "expo-router";
import { Pressable, Text } from "react-native";

export default function HomeScreen() {
  return (
    <>
      {/* Declarative navigation — fully typed */}
      <Link href="/post/42" asChild>
        <Pressable>
          <Text>View Post 42</Text>
        </Pressable>
      </Link>

      {/* Imperative navigation */}
      <Pressable onPress={() => router.push("/post/42")}>
        <Text>Go to Post 42</Text>
      </Pressable>
    </>
  );
}
```

Reading dynamic params in the target screen:

```typescript
// app/post/[id].tsx
import { useLocalSearchParams } from "expo-router";
import { View, Text } from "react-native";
import { useQuery } from "@tanstack/react-query";

export default function PostScreen() {
  const { id } = useLocalSearchParams<{ id: string }>();

  const { data, isLoading } = useQuery({
    queryKey: ["post", id],
    queryFn: () => fetchPost(id),
  });

  if (isLoading) return <Text>Loading...</Text>;

  return (
    <View>
      <Text>{data?.title}</Text>
      <Text>{data?.body}</Text>
    </View>
  );
}
```

NativeWind v4 compiles Tailwind classes to StyleSheet objects at **build time** using Babel. The DX is identical to web Tailwind. No runtime overhead.

```typescript

const styles = StyleSheet.create({
```

## Reanimated 3: Worklets and Gesture Handler

React Native Reanimated 3 runs animations entirely on the **UI thread** using worklets — small JS functions that are serialized and executed in C++, bypassing the JS thread entirely.

### Swipe Card Animation

```typescript
import { View, Text, Dimensions } from "react-native";
import Animated, {
  useSharedValue,
  useAnimatedStyle,
  withSpring,
  runOnJS,
} from "react-native-reanimated";
import {
  Gesture,
  GestureDetector,
} from "react-native-gesture-handler";

const SCREEN_WIDTH = Dimensions.get("window").width;
const SWIPE_THRESHOLD = SCREEN_WIDTH * 0.35;

interface SwipeCardProps {
  title: string;
  onSwipeLeft: () => void;
  onSwipeRight: () => void;
}

export function SwipeCard({ title, onSwipeLeft, onSwipeRight }: SwipeCardProps) {
  const translateX = useSharedValue(0);
  const translateY = useSharedValue(0);
  const rotation = useSharedValue(0);

  const pan = Gesture.Pan()
    .onUpdate((event) => {
      translateX.value = event.translationX;
      translateY.value = event.translationY;
      rotation.value = (event.translationX / SCREEN_WIDTH) * 15;
    })
    .onEnd((event) => {
      if (event.translationX > SWIPE_THRESHOLD) {
        translateX.value = withSpring(SCREEN_WIDTH * 1.5);
        runOnJS(onSwipeRight)();
      } else if (event.translationX < -SWIPE_THRESHOLD) {
        translateX.value = withSpring(-SCREEN_WIDTH * 1.5);
        runOnJS(onSwipeLeft)();
      } else {
        translateX.value = withSpring(0);
        translateY.value = withSpring(0);
        rotation.value = withSpring(0);
      }
    });

  const animatedStyle = useAnimatedStyle(() => ({
    transform: [
      { translateX: translateX.value },
      { translateY: translateY.value },
      { rotate: `${rotation.value}deg` },
    ],
  }));

  return (
    <GestureDetector gesture={pan}>
      <Animated.View
        className="w-80 h-96 bg-white rounded-3xl shadow-xl items-center justify-center"
        style={animatedStyle}
      >
        <Text className="text-2xl font-bold text-zinc-800">{title}</Text>
      </Animated.View>
    </GestureDetector>
  );
}
```

### Interpolating Opacity for Like/Dislike Labels

```typescript
import {
  useAnimatedStyle,
  interpolate,
  Extrapolation,
} from "react-native-reanimated";

function useLikeStyle(translateX: Animated.SharedValue<number>) {
  return useAnimatedStyle(() => ({
    opacity: interpolate(
      translateX.value,
      [0, SWIPE_THRESHOLD],
      [0, 1],
      Extrapolation.CLAMP
    ),
  }));
}

function useDislikeStyle(translateX: Animated.SharedValue<number>) {
  return useAnimatedStyle(() => ({
    opacity: interpolate(
      translateX.value,
      [-SWIPE_THRESHOLD, 0],
      [1, 0],
      Extrapolation.CLAMP
    ),
  }));
}
```

## Shared Element Transitions

Expo Router v3 includes built-in shared element transitions powered by the New Architecture:

```typescript
// In the source screen (list item)
import { SharedTransition, withSpring } from "react-native-reanimated";

const customTransition = SharedTransition.custom((values) => {
  "worklet";
  return {
    height: withSpring(values.targetHeight),
    width: withSpring(values.targetWidth),
    originX: withSpring(values.targetOriginX),
    originY: withSpring(values.targetOriginY),

export function PostCard({ post }: { post: Post }) {
  return (
    <Link href={`/post/${post.id}`} asChild>

        <Animated.Image
```

## EAS Update: OTA Deployments

Expo Application Services (EAS) lets you push JavaScript updates to production without going through the App Store review process. Critical bug fixes can be live in minutes.

### Setup

```bash
npm install -g eas-cli
eas login
eas update:configure
```

This creates an `eas.json`:

```json
{
  "cli": {
    "version": ">= 7.0.0"
  },
  "build": {
    "development": {
      "developmentClient": true,
      "distribution": "internal"
    },
    "preview": {
      "distribution": "internal",
      "channel": "preview"
    },
    "production": {
      "channel": "production",
      "autoIncrement": true
    }
  },
  "update": {
    "channel": "production"
  }
}
```

### Pushing an OTA Update

```bash
# Push to production channel
eas update --branch production --message "Fix checkout crash on Android"

# Preview before pushing
eas update --branch preview --message "Test new home screen layout"
```

### Rollback

```bash
# List recent updates
eas update:list --branch production

# Rollback to a previous update ID
eas update:republish --group <update-group-id> --branch production
```

## Migration Tips from Old Architecture

If you are migrating an existing app, follow these steps:

1. **Enable New Architecture incrementally.** Set `newArchEnabled: true` in `app.json` and run a fresh native build. Fix native module errors one at a time.

2. **Audit third-party libraries.** Check reactnative.directory for the New Architecture badge. Most popular libraries (React Navigation, MMKV, Reanimated, Gesture Handler) are already compatible.

3. **Replace deprecated APIs.** `NativeModules.X` should become typed TurboModule specs. `requireNativeComponent` should migrate to Fabric components with `codegenNativeComponent`.

4. **Remove InteractionManager workarounds.** Many old patterns used `InteractionManager.runAfterInteractions` to defer heavy work. With Concurrent Mode, React itself handles prioritization — these workarounds often cause more harm than good.

5. **Test on physical devices.** The Hermes JS engine and Fabric renderer behave differently from Chrome DevTools. Always profile on real hardware before shipping.

6. **Enable Strict Mode.** Wrap your app in `<React.StrictMode>` to catch concurrent-unsafe effects early.
