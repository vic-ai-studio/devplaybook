Distributed systems are hard to debug. A single user request might touch 8 microservices, 3 databases, and 2 external APIs. When something breaks at 3am, you need to trace that request across every hop instantly. OpenTelemetry (OTel) is the industry standard for making this possible — and in 2026, it's mature enough to deploy with confidence. This guide covers everything from SDK setup to production-ready observability.

## What Is OpenTelemetry?

OpenTelemetry is a CNCF project that provides vendor-neutral APIs, SDKs, and tooling for capturing **traces**, **metrics**, and **logs** — collectively called the "three pillars of observability."

The key advantage: instrument your code once, export to any backend (Jaeger, Prometheus, Grafana Tempo, Datadog, New Relic, etc.) without changing your instrumentation code.

```
┌─────────────────────────────────────────────────────────┐
│                 Your Application                         │
│   ┌─────────────────────────────────────────────────┐   │
│   │           OTel SDK (auto + manual)              │   │
│   └─────────────────┬───────────────────────────────┘   │
└─────────────────────┼───────────────────────────────────┘
                      │ OTLP (gRPC/HTTP)
┌─────────────────────▼───────────────────────────────────┐
│              OTel Collector                              │
│  (receive → process → batch → export)                   │
└─────────┬───────────┬──────────────────┬────────────────┘
          │           │                  │
    ┌─────▼──┐  ┌─────▼──┐       ┌──────▼─────┐
    │ Jaeger │  │ Prom.  │       │   Grafana  │
    │(traces)│  │(metrics│       │   Tempo    │
    └────────┘  └────────┘       └────────────┘
```

## The Three Signals

**Traces** track the journey of a single request across your distributed system. Each trace contains spans — individual operations with start/end times, attributes, and parent-child relationships.

**Metrics** are numeric measurements aggregated over time: request rates, error percentages, latency histograms, queue depths.

**Logs** are timestamped text records. OTel correlates logs with traces via `trace_id` and `span_id` fields.

The magic happens when you **correlate** them: click on a spike in your error rate metric → see the affected traces → jump to the logs for a specific failing span.

### Auto-Instrumentation (Recommended Starting Point)

Create an instrumentation file loaded before your app:

```typescript
// instrumentation.ts
import { NodeSDK } from "@opentelemetry/sdk-node";
import { getNodeAutoInstrumentations } from "@opentelemetry/auto-instrumentations-node";
import { OTLPTraceExporter } from "@opentelemetry/exporter-trace-otlp-proto";
import { OTLPMetricExporter } from "@opentelemetry/exporter-metrics-otlp-proto";
import { PeriodicExportingMetricReader } from "@opentelemetry/sdk-metrics";
import { Resource } from "@opentelemetry/resources";
import { ATTR_SERVICE_NAME, ATTR_SERVICE_VERSION } from "@opentelemetry/semantic-conventions";

const sdk = new NodeSDK({
  resource: new Resource({
    [ATTR_SERVICE_NAME]: "payment-service",
    [ATTR_SERVICE_VERSION]: "1.2.0",
    "deployment.environment": process.env.NODE_ENV ?? "development",
  }),

  traceExporter: new OTLPTraceExporter({
    url: "http://otel-collector:4318/v1/traces",
  }),

  metricReader: new PeriodicExportingMetricReader({
    exporter: new OTLPMetricExporter({
      url: "http://otel-collector:4318/v1/metrics",
    }),
    exportIntervalMillis: 15000,
  }),

  instrumentations: [
    getNodeAutoInstrumentations({
      "@opentelemetry/instrumentation-http": { enabled: true },
      "@opentelemetry/instrumentation-express": { enabled: true },
      "@opentelemetry/instrumentation-pg": { enabled: true },
      "@opentelemetry/instrumentation-redis": { enabled: true },
      "@opentelemetry/instrumentation-grpc": { enabled: true },
      // Disable noisy instrumentations
      "@opentelemetry/instrumentation-fs": { enabled: false },
    }),
  ],
});

sdk.start();

process.on("SIGTERM", () => {
  sdk.shutdown().then(() => process.exit(0));
});
```

Start your app with the instrumentation loaded:

```bash
node --require ./instrumentation.js src/server.js
# Or with tsx:
tsx --require ./instrumentation.ts src/server.ts
```

Auto-instrumentation automatically traces all HTTP requests, database queries, Redis operations, and gRPC calls — no code changes needed.

### Manual Spans for Business Logic

```typescript
import { trace, SpanStatusCode, SpanKind } from "@opentelemetry/api";

const tracer = trace.getTracer("payment-service", "1.2.0");

async function processPayment(orderId: string, amount: number): Promise<PaymentResult> {
  return tracer.startActiveSpan(
    "payment.process",
    {
      kind: SpanKind.INTERNAL,
      attributes: {
        "payment.order_id": orderId,
        "payment.amount": amount,
        "payment.currency": "USD",
      },
    },
    async (span) => {
      try {
        // Nested spans are automatically children
        const validation = await validatePayment(orderId, amount);
        span.addEvent("payment.validated", { "validation.result": validation.status });

        const charge = await chargeCard(validation.cardToken, amount);
        span.setAttributes({
          "payment.transaction_id": charge.transactionId,
          "payment.provider": charge.provider,
        });

        span.setStatus({ code: SpanStatusCode.OK });
        return { success: true, transactionId: charge.transactionId };
      } catch (error) {
        span.recordException(error as Error);
        span.setStatus({
          code: SpanStatusCode.ERROR,
          message: (error as Error).message,
        });
        throw error;
      } finally {
        span.end();
      }
    }
  );
}
```

### Custom Metrics

```typescript
import { metrics } from "@opentelemetry/api";

const meter = metrics.getMeter("payment-service");

// Counter: monotonically increasing
const paymentCounter = meter.createCounter("payments.total", {
  description: "Total number of payment attempts",
});

// Histogram: for latency distributions
const paymentDuration = meter.createHistogram("payments.duration", {
  description: "Payment processing duration in ms",
  unit: "ms",
  boundaries: [10, 50, 100, 250, 500, 1000, 2500, 5000],
});

// UpDownCounter: can go up or down
const activePayments = meter.createUpDownCounter("payments.active", {
  description: "Currently processing payments",
});

// Observable Gauge: measured externally
const queueDepthGauge = meter.createObservableGauge("payments.queue.depth", {
  description: "Payment queue depth",
});

queueDepthGauge.addCallback(async (result) => {
  const depth = await getQueueDepth();
  result.observe(depth, { "queue.name": "payment-processing" });
});

// Usage
async function processPaymentWithMetrics(orderId: string, amount: number) {
  const startTime = Date.now();
  activePayments.add(1, { "payment.type": "credit_card" });

  try {
    const result = await processPayment(orderId, amount);
    paymentCounter.add(1, { "payment.status": "success" });
    return result;
  } catch (error) {
    paymentCounter.add(1, { "payment.status": "failure" });
    throw error;
  } finally {
    paymentDuration.record(Date.now() - startTime);
    activePayments.add(-1, { "payment.type": "credit_card" });
  }
}
```

### FastAPI Auto-Instrumentation

```python
# instrumentation.py
from opentelemetry import trace, metrics
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.sdk.metrics import MeterProvider
from opentelemetry.sdk.metrics.export import PeriodicExportingMetricReader
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.exporter.otlp.proto.grpc.metric_exporter import OTLPMetricExporter
from opentelemetry.sdk.resources import Resource
from opentelemetry.semconv.resource import ResourceAttributes

def setup_telemetry(service_name: str, service_version: str):
    resource = Resource.create({
        ResourceAttributes.SERVICE_NAME: service_name,
        ResourceAttributes.SERVICE_VERSION: service_version,
        ResourceAttributes.DEPLOYMENT_ENVIRONMENT: os.getenv("ENVIRONMENT", "development"),
    })

    # Traces
    tracer_provider = TracerProvider(resource=resource)
    tracer_provider.add_span_processor(
        BatchSpanProcessor(
            OTLPSpanExporter(endpoint="http://otel-collector:4317")
        )
    )
    trace.set_tracer_provider(tracer_provider)

    # Metrics
    metric_reader = PeriodicExportingMetricReader(
        OTLPMetricExporter(endpoint="http://otel-collector:4317"),
        export_interval_millis=15000,
    )
    meter_provider = MeterProvider(resource=resource, metric_readers=[metric_reader])
    metrics.set_meter_provider(meter_provider)
```

```python
# main.py
from fastapi import FastAPI, Request
from opentelemetry.instrumentation.fastapi import FastAPIInstrumentor
from opentelemetry.instrumentation.sqlalchemy import SQLAlchemyInstrumentor
from opentelemetry.instrumentation.redis import RedisInstrumentor
from opentelemetry import trace, metrics
from instrumentation import setup_telemetry

setup_telemetry("user-service", "2.1.0")

app = FastAPI()

# Auto-instrument FastAPI, SQLAlchemy, and Redis
FastAPIInstrumentor.instrument_app(app)
SQLAlchemyInstrumentor().instrument(engine=engine)
RedisInstrumentor().instrument()

tracer = trace.get_tracer("user-service")
meter = metrics.get_meter("user-service")

# Custom metrics
request_counter = meter.create_counter(
    "api.requests",
    description="Total API requests"
)

@app.get("/users/{user_id}")
async def get_user(user_id: str, request: Request):
    request_counter.add(1, {
        "http.method": "GET",
        "http.route": "/users/{user_id}"
    })

    with tracer.start_as_current_span("user.fetch") as span:
        span.set_attribute("user.id", user_id)

        user = await db.users.get(user_id)
        if not user:
            span.set_attribute("user.found", False)
            raise HTTPException(status_code=404)

        span.set_attribute("user.found", True)
        span.set_attribute("user.email_domain", user.email.split("@")[1])
        return user
```

## Correlating Traces, Metrics, and Logs

The killer feature: linking all three signals together. Add trace context to logs:

### Node.js Log Correlation

```typescript
import { trace } from "@opentelemetry/api";
import winston from "winston";

const logger = winston.createLogger({
  format: winston.format.combine(
    winston.format.timestamp(),
    winston.format.json(),
    // Add OTel trace context to every log entry
    winston.format((info) => {
      const activeSpan = trace.getActiveSpan();
      if (activeSpan) {
        const ctx = activeSpan.spanContext();
        info["trace_id"] = ctx.traceId;
        info["span_id"] = ctx.spanId;
        info["trace_flags"] = ctx.traceFlags;
      }
      return info;
    })()
  ),
  transports: [new winston.transports.Console()],
});

// Every log now contains trace_id — correlate in Grafana with a single click
logger.info("Payment processed", {
  orderId: "ORD-123",
  amount: 99.99,
});
```

Output:
```json

  "timestamp": "2026-03-28T10:00:00.000Z",

  "message": "Payment processed",
```

## OTel Collector Configuration

The Collector receives telemetry, processes it, and routes to backends. Deploy as a sidecar or standalone:

```yaml
# otel-collector-config.yaml
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318

processors:
  batch:
    timeout: 5s
    send_batch_size: 1000

  # Add resource attributes to all telemetry
  resource:
    attributes:
      - key: "cluster.name"
        value: "production-us-east-1"
        action: insert

  # Tail-based sampling: keep 100% of error traces, 10% of success
  tail_sampling:
    decision_wait: 10s
    num_traces: 100000
    policies:
      - name: errors-policy
        type: status_code
        status_code: { status_codes: [ERROR] }
      - name: slow-traces-policy
        type: latency
        latency: { threshold_ms: 1000 }
      - name: sample-rest
        type: probabilistic
        probabilistic: { sampling_percentage: 10 }

exporters:
  jaeger:
    endpoint: jaeger:14250
    tls:
      insecure: true

  prometheus:
    endpoint: "0.0.0.0:8889"

  otlp/tempo:
    endpoint: tempo:4317
    tls:
      insecure: true

  loki:
    endpoint: http://loki:3100/loki/api/v1/push

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch, resource, tail_sampling]
      exporters: [jaeger, otlp/tempo]

    metrics:
      receivers: [otlp]
      processors: [batch, resource]
      exporters: [prometheus]

    logs:
      receivers: [otlp]
      processors: [batch, resource]
      exporters: [loki]
```

## Grafana Dashboard Setup

Grafana is the go-to visualization layer. Configure datasources:

```yaml
# grafana/provisioning/datasources/otel.yaml
apiVersion: 1
datasources:
  - name: Jaeger
    type: jaeger
    url: http://jaeger:16686
    jsonData:
      tracesToLogsV2:
        datasourceUid: loki
        spanStartTimeShift: "-1m"
        spanEndTimeShift: "1m"
        filterByTraceID: true
        filterBySpanID: true

  - name: Prometheus
    type: prometheus
    url: http://prometheus:9090
    jsonData:
      exemplarTraceIdDestinations:
        - name: trace_id
          datasourceUid: tempo

  - name: Tempo
    type: tempo
    url: http://tempo:3200
    jsonData:
      tracesToLogs:
        datasourceUid: loki
```

With this setup, you can:
1. See an error spike in Prometheus → click the error rate panel
2. Jump to an exemplar trace (a real request that caused the error)
3. From the trace, jump to the correlated logs for that exact request

## Production Best Practices

**Sampling strategy:** Never sample 100% in production. Use tail-based sampling (keep errors + slow traces, sample the rest). A good starting point: 100% errors, 100% traces > 2s, 5% of successful fast traces.

**Semantic conventions:** Use OTel semantic conventions for attribute names (`http.method`, `db.statement`, `rpc.service`). Backends like Grafana provide automatic dashboards for services that follow conventions.

**Cardinality control:** Never use user IDs, request IDs, or other high-cardinality values as metric labels. Use spans/traces for per-request data; use metrics for aggregated data.

```typescript
// BAD: High cardinality metric label
paymentCounter.add(1, { "payment.user_id": userId });  // Creates millions of series

// GOOD: Low cardinality metric label
paymentCounter.add(1, { "payment.type": "credit_card", "payment.status": "success" });

// HIGH CARDINALITY: Goes in span attributes (not metric labels)
span.setAttributes({ "payment.user_id": userId });
```

**Resource overhead:** OTel SDK adds ~5-15ms to first request (SDK initialization) and <1ms per span. The Collector handles batching to minimize network overhead.

## Getting Started Quickly

Use the OTel Docker Compose starter:

```yaml
# docker-compose.yaml (development)
services:
  otel-collector:
    image: otel/opentelemetry-collector-contrib:latest
    volumes:
      - ./otel-collector-config.yaml:/etc/otelcol-contrib/config.yaml

      - "4317:4317"   # gRPC

  jaeger:
    image: jaegertracing/all-in-one:latest
    ports:
      - "16686:16686"   # UI

  prometheus:
    image: prom/prometheus:latest
    ports:
      - "9090:9090"

  grafana:
    image: grafana/grafana:latest
    ports:
      - "3000:3000"
    environment:
      - GF_AUTH_ANONYMOUS_ENABLED=true
```

```bash
docker compose up -d
# OTel Collector: localhost:4317
# Jaeger UI: localhost:16686
# Grafana: localhost:3000
```
