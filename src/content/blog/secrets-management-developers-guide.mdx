# Secrets Management for Developers: Vault, Doppler & .env Best Practices

Every application has secrets: database passwords, API keys, JWT signing keys, OAuth credentials. How you manage these secrets — storing, rotating, distributing, and auditing them — is one of the most important security decisions you make.

---

## The Problem with .env Files

`.env` files are convenient but dangerous when misused:

```bash
# .env — fine for local dev, NEVER commit to git
DATABASE_URL=postgres://user:password@localhost/mydb
STRIPE_SECRET_KEY=sk_live_abc123xyz
JWT_SECRET=my-super-secret

# The disaster pattern:
git add .
git commit -m "Add env config"
git push origin main  # Secret is now in your git history FOREVER
```

Even if you delete it later, the secret exists in git history. Tools like `git-secrets`, Gitleaks, and GitHub's push protection scan for this.

### .env Rules

```bash
# .gitignore — ALWAYS include
.env
.env.local
.env.production
.env.*.local

# Commit this safe template instead
.env.example  # Contains keys but NOT values
```

```bash
# .env.example — safe to commit
DATABASE_URL=
STRIPE_SECRET_KEY=
JWT_SECRET=
```

## Local Development: Secrets Without Committing

### Option 1: 1Password CLI

```bash
# Store secrets in 1Password, inject at runtime
eval $(op signin)
op run --env-file=.env.tpl -- npm run dev

# .env.tpl (safe to commit — references 1Password items)
DATABASE_URL=op://Personal/MyApp Dev/database_url
STRIPE_SECRET_KEY=op://Personal/Stripe Dev/secret_key
```

### Option 2: Doppler (Cloud Secrets Manager)

```bash
# Install Doppler CLI
brew install dopplerhq/cli/doppler

# Setup once per project
doppler setup

# Inject secrets into your dev process
doppler run -- npm run dev
doppler run -- node server.js
```

Doppler syncs secrets from its dashboard to your dev environment without storing anything locally. Changes propagate instantly.

## Production: Secrets Managers

### AWS Secrets Manager

```javascript
import { SecretsManagerClient, GetSecretValueCommand } from "@aws-sdk/client-secrets-manager";

const client = new SecretsManagerClient({ region: "us-east-1" });

async function getSecret(secretName: string): Promise<Record<string, string>> {
  const response = await client.send(
    new GetSecretValueCommand({ SecretId: secretName })
  );
  return JSON.parse(response.SecretString || '{}');
}

// Cache secrets at startup (don't call on every request!)
let cachedSecrets: Record<string, string>;

async function getDbConfig() {
  if (!cachedSecrets) {
    cachedSecrets = await getSecret('myapp/production/database');
  }
  return {
    host: cachedSecrets.host,
    password: cachedSecrets.password,
    database: cachedSecrets.database,
  };
}
```

### HashiCorp Vault

```bash
# Write a secret
vault kv put secret/myapp/database \
  host=db.production.internal \
  password=supersecretpassword

# Read in application
vault kv get -format=json secret/myapp/database
```

```javascript
import vault from 'node-vault';

const client = vault({
  apiVersion: 'v1',
  endpoint: process.env.VAULT_ADDR,
  token: process.env.VAULT_TOKEN,  // Use AppRole or AWS auth in production
});

const secrets = await client.read('secret/data/myapp/database');
const { host, password } = secrets.data.data;
```

### Kubernetes: External Secrets Operator

```yaml
# ExternalSecret — syncs from AWS Secrets Manager to a K8s Secret
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: database-credentials
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-secrets-manager
    kind: SecretStore
  target:
    name: database-credentials  # Creates this K8s Secret
  data:
    - secretKey: password
      remoteRef:
        key: myapp/production/database
        property: password
```

## CI/CD Secret Injection

### GitHub Actions

```yaml
# Store in GitHub Settings > Secrets and Variables > Actions
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Deploy
        env:
          DATABASE_URL: ${{ secrets.DATABASE_URL }}
          STRIPE_SECRET: ${{ secrets.STRIPE_SECRET_KEY }}
        run: ./deploy.sh
```

Never echo secrets in CI logs:
```yaml
# BAD: Secret visible in logs
- run: echo "DB URL is $DATABASE_URL"

# GOOD: GitHub masks registered secrets automatically, but don't print them
- run: |
    if [ -z "$DATABASE_URL" ]; then
      echo "ERROR: DATABASE_URL not set"
      exit 1
    fi
```

### Docker: Build-time vs Runtime Secrets

```dockerfile
# BAD: Secret baked into image layer (visible in `docker history`)
ARG API_KEY
ENV API_KEY=$API_KEY

# GOOD: Use Docker BuildKit secrets (not stored in image)
# syntax=docker/dockerfile:1
RUN --mount=type=secret,id=api_key \
    API_KEY=$(cat /run/secrets/api_key) npm run build

# Build:
docker build --secret id=api_key,env=API_KEY .

# BETTER for runtime: Inject at container start, not build time
# Use env vars from secrets manager at runtime:
docker run -e DATABASE_URL=$(aws secretsmanager get-secret-value ...) myapp
```

## Secret Rotation

Hardcoded secrets that never rotate are a liability. Automate rotation:

```javascript
// Pattern: Read secret at startup, re-read on rotation signal
import { SecretsManagerClient, GetSecretValueCommand } from "@aws-sdk/client-secrets-manager";

class RotatingSecret {
  private secret: string;
  private lastFetched: Date;
  private readonly ttl = 3600 * 1000; // 1 hour cache

  constructor(private secretName: string) {}

  async get(): Promise<string> {
    const now = new Date();
    if (!this.secret || now.getTime() - this.lastFetched.getTime() > this.ttl) {
      this.secret = await this.fetchFromManager();
      this.lastFetched = now;
    }
    return this.secret;
  }

  private async fetchFromManager(): Promise<string> {
    const client = new SecretsManagerClient({ region: 'us-east-1' });
    const { SecretString } = await client.send(
      new GetSecretValueCommand({ SecretId: this.secretName })
    );
    return SecretString || '';
  }
}

const dbPassword = new RotatingSecret('myapp/production/db-password');
```

## Auditing: What to Log

```javascript
// Always log secret access for audit trail
const securityLog = {
  secretAccessed: (secretName: string, caller: string) => {
    logger.info({
      event: 'secret.accessed',
      secret: secretName,
      caller,
      timestamp: new Date().toISOString(),
    });
  },
  secretRotated: (secretName: string) => {
    logger.info({ event: 'secret.rotated', secret: secretName });
  },
};
```

AWS Secrets Manager logs all access to CloudTrail automatically. Set up alerts for:
- Unusual access patterns (unexpected time, IP, principal)
- Failed access attempts
- Secret value retrievals from new locations

## Secrets Management Checklist

- [ ] `.env` in `.gitignore`, never committed
- [ ] `.env.example` committed with blank values
- [ ] Local dev: secrets from 1Password CLI, Doppler, or Vault
- [ ] CI/CD: secrets from platform vault (GitHub Secrets, GitLab CI variables)
- [ ] Production: AWS Secrets Manager, HashiCorp Vault, or GCP Secret Manager
- [ ] No secrets in Docker image layers (use runtime injection)
- [ ] Secret rotation automated or scheduled
- [ ] Access audit logging enabled
