# OWASP Top 10 for Developers: Practical Prevention Guide

The OWASP Top 10 is the most referenced list of web application security risks. Understanding it isn't just for security teams — every developer shipping web applications should know these patterns cold. Here's each risk with a vulnerable example and the fix.

---

## A01: Broken Access Control

The most common vulnerability in 2024. It occurs when authorization logic is missing or incorrect.

```javascript
// VULNERABLE: Direct Object Reference without authorization check
app.get('/api/documents/:id', async (req, res) => {
  const doc = await db.documents.findById(req.params.id);
  res.json(doc);  // Returns ANY document regardless of who's asking
});

// SECURE: Check ownership before returning
app.get('/api/documents/:id', authenticate, async (req, res) => {
  const doc = await db.documents.findById(req.params.id);
  if (!doc) return res.status(404).json({ error: 'Not found' });
  if (doc.userId !== req.user.id) return res.status(403).json({ error: 'Forbidden' });
  res.json(doc);
});
```

**Prevention:**
- Deny by default — require explicit permission grants
- Log access control failures and alert on patterns
- Use centralized authorization middleware, not ad-hoc checks

## A02: Cryptographic Failures

Sensitive data exposed due to weak encryption, missing encryption, or storing secrets in plaintext.

```javascript
// VULNERABLE: Storing passwords in plaintext or MD5
const user = await db.users.create({
  email: email,
  password: password,          // plaintext!
  // OR: password: md5(password)  // MD5 is broken
});

// SECURE: bcrypt with appropriate cost factor
import bcrypt from 'bcryptjs';

const hash = await bcrypt.hash(password, 12);  // cost 12 ≈ 250ms on modern CPU
const user = await db.users.create({ email, passwordHash: hash });

// Verification
const valid = await bcrypt.compare(inputPassword, user.passwordHash);
```

**Prevention:**
- Never store plaintext passwords — always bcrypt/Argon2/scrypt
- TLS everywhere, including internal service communication
- Use secrets managers (Vault, AWS Secrets Manager) not environment variables in plaintext

## A03: Injection

SQL injection, command injection, LDAP injection — all share the same root cause: user input is interpreted as code.

```javascript
// VULNERABLE: String interpolation in SQL query
const query = `SELECT * FROM users WHERE email = '${req.body.email}'`;
// Attack: email = "' OR '1'='1" → returns all users
// Attack: email = "'; DROP TABLE users; --" → drops table

// SECURE: Parameterized queries
const user = await db.query('SELECT * FROM users WHERE email = $1', [req.body.email]);

// With ORM (Prisma)
const user = await prisma.users.findUnique({ where: { email: req.body.email } });
```

```javascript
// VULNERABLE: Command injection
app.post('/convert', (req, res) => {
  exec(`convert ${req.body.inputFile} output.png`, callback);  // RCE if inputFile = "; rm -rf /"
});

// SECURE: Never pass user input to shell; use library instead
import sharp from 'sharp';
app.post('/convert', async (req, res) => {
  const buffer = await sharp(sanitizedPath).png().toBuffer();
  res.send(buffer);
});
```

## A04: Insecure Design

Architectural flaws that no amount of correct implementation can fix — missing rate limiting, no account lockout, insecure password reset flows.

```javascript
// VULNERABLE: Password reset with predictable token
const token = Date.now().toString();  // Predictable! Brute-forceable.
const token2 = user.id + user.email;  // Also predictable

// SECURE: Cryptographically random token with expiry
import crypto from 'node:crypto';

const token = crypto.randomBytes(32).toString('hex');  // 256 bits, unpredictable
const expiry = new Date(Date.now() + 3600 * 1000);     // 1 hour expiry

await db.passwordResets.create({
  userId: user.id,
  token: token,  // hash this before storing if you want extra safety
  expiresAt: expiry,
  used: false,
});
```

## A05: Security Misconfiguration

Default credentials, verbose error messages, missing security headers, open cloud storage.

```javascript
// VULNERABLE: Exposing stack traces to users
app.use((err, req, res, next) => {
  res.status(500).json({ error: err.stack });  // Exposes internal details
});

// SECURE: Log internally, return generic message
app.use((err, req, res, next) => {
  logger.error({ err, url: req.url, userId: req.user?.id });
  res.status(500).json({ error: 'An unexpected error occurred.' });
});
```

```javascript
// Security headers with helmet
import helmet from 'helmet';

app.use(helmet({
  contentSecurityPolicy: {
    directives: {
      defaultSrc: ["'self'"],
      scriptSrc: ["'self'"],
    },
  },
  hsts: { maxAge: 31536000, includeSubDomains: true },
  referrerPolicy: { policy: 'strict-origin-when-cross-origin' },
}));
```

## A06: Vulnerable and Outdated Components

Using npm packages with known CVEs.

```bash
# Check for vulnerabilities
npm audit
npm audit --audit-level=high  # fail on high+ severity

# Auto-fix safe upgrades
npm audit fix

# Use Snyk for deeper analysis
npx snyk test
```

Set up Dependabot in `.github/dependabot.yml`:
```yaml
version: 2
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
    open-pull-requests-limit: 10
```

## A07: Identification and Authentication Failures

Weak passwords, missing MFA, improper session management.

```javascript
// VULNERABLE: JWT with weak secret, no expiry
const token = jwt.sign({ userId: user.id }, 'secret');

// SECURE: Strong secret, short expiry, proper algorithm
const token = jwt.sign(
  { userId: user.id, role: user.role },
  process.env.JWT_SECRET,  // 256-bit random secret from secrets manager
  {
    algorithm: 'HS256',
    expiresIn: '15m',    // Short-lived access token
    issuer: 'api.example.com',
  }
);

// Implement refresh token rotation for persistent sessions
```

## A08: Software and Data Integrity Failures

Deserialization attacks, CI/CD pipeline compromise, auto-updates without integrity checks.

```bash
# Use lockfiles and verify integrity
npm ci  # Installs exact versions from package-lock.json (not npm install)

# Enable npm audit in CI
npm audit --audit-level=moderate

# Verify package integrity with Sigstore (emerging standard)
```

## A09: Security Logging and Monitoring Failures

No audit trail for authentication events, no alerting on suspicious patterns.

```javascript
// Log security-relevant events
const securityLogger = {
  authSuccess: (userId: string, ip: string) =>
    logger.info({ event: 'auth.success', userId, ip }),

  authFailure: (email: string, ip: string, reason: string) =>
    logger.warn({ event: 'auth.failure', email, ip, reason }),

  accessDenied: (userId: string, resource: string, action: string) =>
    logger.warn({ event: 'access.denied', userId, resource, action }),

  suspiciousActivity: (detail: object) =>
    logger.error({ event: 'suspicious', ...detail }),
};

// Alert on 5+ failed logins from same IP in 5 minutes
// Alert on admin actions
// Alert on mass data export
```

## A10: Server-Side Request Forgery (SSRF)

An attacker causes the server to make HTTP requests to internal services.

```javascript
// VULNERABLE: Fetching user-supplied URL without validation
app.post('/preview', async (req, res) => {
  const content = await fetch(req.body.url).then(r => r.text());
  // Attack: url = "http://169.254.169.254/latest/meta-data/" (AWS metadata)
  // Attack: url = "http://localhost:6379/" (Redis)
});

// SECURE: Validate against allowlist, block private IPs
import { URL } from 'url';

function isSafeUrl(urlStr: string): boolean {
  try {
    const url = new URL(urlStr);
    if (!['http:', 'https:'].includes(url.protocol)) return false;
    const hostname = url.hostname;
    // Block private IP ranges
    if (/^(10\.|172\.(1[6-9]|2\d|3[01])\.|192\.168\.|127\.|169\.254\.|::1)/.test(hostname)) return false;
    // Only allow known domains
    const allowlist = ['example.com', 'api.example.com'];
    return allowlist.some(d => hostname === d || hostname.endsWith('.' + d));
  } catch { return false; }
}
```

## Quick Reference

| OWASP Risk | Key Prevention |
|---|---|
| A01 Broken Access Control | Deny by default, centralized authz |
| A02 Cryptographic Failures | bcrypt passwords, TLS everywhere |
| A03 Injection | Parameterized queries, no shell commands |
| A04 Insecure Design | Threat modeling, rate limiting |
| A05 Misconfiguration | Security headers, no stack traces |
| A06 Vulnerable Components | npm audit, Dependabot |
| A07 Auth Failures | MFA, short-lived JWTs, lockout |
| A08 Integrity Failures | npm ci, lockfiles, signed commits |
| A09 Logging Failures | Log auth events, alert on patterns |
| A10 SSRF | URL allowlist, block private IPs |
