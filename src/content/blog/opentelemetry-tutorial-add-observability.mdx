OpenTelemetry (OTel) is the open standard for observability. It instruments your application once and sends traces, metrics, and logs to any backend—Jaeger, Grafana Tempo, Datadog, New Relic, Honeycomb—without vendor lock-in. This tutorial gets you fully instrumented in 15 minutes.

## The Three Pillars: Traces, Metrics, Logs

**Traces** show the full journey of a request through your system. A single trace contains spans—one span per service, database call, or external request. Traces answer: "Why was this request slow?"

**Metrics** are numerical measurements over time: request rate, error rate, latency percentiles, queue depth. Metrics answer: "What is happening right now?"

**Logs** are timestamped records of discrete events. Logs answer: "What happened exactly?"

OpenTelemetry's power is correlating all three: click a high-latency spike in your metrics, drill into a trace for that time window, find the slow span, click through to the logs from that span.

## Set Up Automatic Instrumentation

Create `instrumentation.js` (or `instrumentation.ts`). This must be loaded **before** your app code:

```javascript
// instrumentation.js
import { NodeSDK } from '@opentelemetry/sdk-node';
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node';
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-http';
import { OTLPMetricExporter } from '@opentelemetry/exporter-metrics-otlp-http';
import { PeriodicExportingMetricReader } from '@opentelemetry/sdk-metrics';
import { Resource } from '@opentelemetry/resources';
import { SEMRESATTRS_SERVICE_NAME, SEMRESATTRS_SERVICE_VERSION } from '@opentelemetry/semantic-conventions';

const sdk = new NodeSDK({
  resource: new Resource({
    [SEMRESATTRS_SERVICE_NAME]: 'payment-service',
    [SEMRESATTRS_SERVICE_VERSION]: '1.0.0',
    'deployment.environment': process.env.NODE_ENV || 'development',
  }),

  // Trace exporter — sends to OTLP endpoint (Jaeger, Tempo, Collector, etc.)
  traceExporter: new OTLPTraceExporter({
    url: process.env.OTEL_EXPORTER_OTLP_ENDPOINT || 'http://localhost:4318/v1/traces',
  }),

  // Metrics exporter
  metricReader: new PeriodicExportingMetricReader({
    exporter: new OTLPMetricExporter({
      url: process.env.OTEL_EXPORTER_OTLP_ENDPOINT || 'http://localhost:4318/v1/metrics',
    }),
    exportIntervalMillis: 30000, // every 30 seconds
  }),

  // Auto-instrument popular libraries automatically
  instrumentations: [
    getNodeAutoInstrumentations({
      '@opentelemetry/instrumentation-http': { enabled: true },
      '@opentelemetry/instrumentation-express': { enabled: true },
      '@opentelemetry/instrumentation-pg': { enabled: true },
      '@opentelemetry/instrumentation-redis': { enabled: true },
      '@opentelemetry/instrumentation-fs': { enabled: false }, // noisy, usually disable
    }),
  ],
});

sdk.start();
console.log('OpenTelemetry instrumentation started');

// Graceful shutdown
process.on('SIGTERM', () => sdk.shutdown());
```

Load it before your app:

```json
// package.json
{
  "scripts": {
    "start": "node --require ./instrumentation.js src/server.js"
  }
}
```

That's it for auto-instrumentation. HTTP requests, Express routes, database queries, and Redis calls are all traced automatically.

## What Auto-Instrumentation Gives You

With the setup above, you immediately get:
- Every HTTP request as a trace with status code, URL, method
- Every Express route handler as a child span
- PostgreSQL queries with SQL statements (configurable)
- Redis commands with keys
- HTTP calls to external services
- Automatic error recording on exceptions

Start your app, make a few requests, and open Jaeger at `http://localhost:16686`. You'll see traces without writing a single span manually.

## Manual Instrumentation: Adding Custom Spans

Auto-instrumentation captures infrastructure. Manual spans capture your business logic:

```javascript
// src/services/payment.js
import { trace, SpanStatusCode } from '@opentelemetry/api';

const tracer = trace.getTracer('payment-service');

export async function processPayment(orderId, amount, currency) {
  // Create a span for this business operation
  return tracer.startActiveSpan('process_payment', async (span) => {
    try {
      // Add business context as attributes
      span.setAttributes({
        'payment.order_id': orderId,
        'payment.amount': amount,
        'payment.currency': currency,
      });

      // Validate
      span.addEvent('validation_started');
      await validatePaymentDetails(orderId, amount);
      span.addEvent('validation_completed');

      // Charge
      const result = await chargeCard(amount, currency);
      span.setAttribute('payment.transaction_id', result.transactionId);

      // Record success
      span.setStatus({ code: SpanStatusCode.OK });
      return result;

    } catch (error) {
      // Record the error on the span
      span.recordException(error);
      span.setStatus({ code: SpanStatusCode.ERROR, message: error.message });
      throw error;

    } finally {
      span.end(); // always end the span
    }
  });
}
```

## Adding Custom Metrics

```javascript
// src/metrics.js
import { metrics } from '@opentelemetry/api';

const meter = metrics.getMeter('payment-service');

// Counter: things that happen (payments processed, errors)
export const paymentsProcessed = meter.createCounter('payments.processed', {
  description: 'Total number of payments processed',
  unit: '1',
});

// Histogram: distribution of values (latency, amount)
export const paymentDuration = meter.createHistogram('payments.duration', {
  description: 'Payment processing duration',
  unit: 'ms',
});

// UpDownCounter: values that go up and down (queue depth, connections)
export const activeConnections = meter.createUpDownCounter('db.connections.active', {
  description: 'Number of active database connections',
});

// Usage in your service:
const start = Date.now();
await processPayment(orderId, amount, currency);
const duration = Date.now() - start;

paymentsProcessed.add(1, { currency, status: 'success' });
paymentDuration.record(duration, { currency });
```

## Context Propagation

Context propagation is what makes distributed tracing work across services. When Service A calls Service B, the trace context (trace ID, span ID) is passed in HTTP headers so both spans appear in the same trace.

Auto-instrumentation handles this automatically for HTTP calls. But for custom message passing or job queues, you need to propagate manually:

```javascript
// Service A: inject context into message queue payload
import { context, propagation } from '@opentelemetry/api';

async function publishToQueue(queue, payload) {
  const carrier = {};
  propagation.inject(context.active(), carrier); // inject trace context

  await queue.publish({
    ...payload,
    _otel: carrier, // include context in message
  });
}

// Service B: extract context from message queue payload
async function consumeFromQueue(message) {
  const carrier = message._otel || {};
  const ctx = propagation.extract(context.active(), carrier); // extract trace context

  return context.with(ctx, async () => {
    // All spans created here will be linked to the original trace
    return tracer.startActiveSpan('process_queue_message', async (span) => {
      await processMessage(message);
      span.end();
    });
  });
}
```

## Auto vs Manual Instrumentation: When to Use Each

| Use Auto-Instrumentation For | Use Manual Instrumentation For |
|---|---|
| HTTP servers and clients | Business logic operations |
| Database queries (pg, mysql, mongo) | Queue message processing |
| Redis operations | External API integrations |
| gRPC calls | Batch processing jobs |
| Popular frameworks (Express, Fastapi) | Custom caching layers |

The rule of thumb: if a library instrumentation package exists, use it. Add manual spans for your own business logic and anything that doesn't have a library.

## Sending to Different Backends

The OTLP (OpenTelemetry Protocol) exporter works with any compatible backend. Change the endpoint URL:

```bash
# Jaeger
OTEL_EXPORTER_OTLP_ENDPOINT=http://jaeger:4318

# Grafana Tempo
OTEL_EXPORTER_OTLP_ENDPOINT=http://tempo:4318

# Datadog (via Datadog Agent)
OTEL_EXPORTER_OTLP_ENDPOINT=http://datadog-agent:4318

# Honeycomb
OTEL_EXPORTER_OTLP_ENDPOINT=https://api.honeycomb.io
OTEL_EXPORTER_OTLP_HEADERS=x-honeycomb-team=YOUR_API_KEY

# New Relic
OTEL_EXPORTER_OTLP_ENDPOINT=https://otlp.nr-data.net
OTEL_EXPORTER_OTLP_HEADERS=api-key=YOUR_LICENSE_KEY
```

No code changes needed—just environment variables.

## Production Checklist

Before shipping OTel to production:

- [ ] Set `OTEL_SAMPLING_RATIO=0.1` to sample 10% of traces (adjust for volume)
- [ ] Never log secrets or PII in span attributes
- [ ] Disable filesystem instrumentation (`@opentelemetry/instrumentation-fs`) — extremely noisy
- [ ] Set `service.name` and `deployment.environment` in resource attributes
- [ ] Configure span attribute limits to prevent memory bloat
- [ ] Use async exporter (default) — never block request handling on span export
- [ ] Implement health check that verifies OTel SDK started successfully

OpenTelemetry takes 15 minutes to set up and gives you years of production debugging capability. Start with auto-instrumentation, add manual spans for your critical business flows, and you'll never fly blind in production again.
