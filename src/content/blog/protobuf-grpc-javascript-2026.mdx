When services need to communicate efficiently at scale, JSON over HTTP starts to show its limits. Protocol Buffers (protobuf) and gRPC offer a compelling alternative: strongly-typed, compact binary encoding, automatic code generation, and built-in support for streaming. In 2026, the JavaScript tooling has matured enough that gRPC is a practical choice for serious projects.

## What Is Protobuf vs JSON?

Protocol Buffers is a language-neutral, platform-neutral mechanism for serializing structured data. Think of it as a schema-first approach to data serialization with automatic code generation.

### Size and Speed

```
JSON (typical API response):
{"userId": 123, "name": "Alice", "email": "alice@example.com", "active": true}
= 70 bytes

Protobuf equivalent:
[binary encoding]
= ~30 bytes (57% smaller)
```

The binary encoding also parses faster — no string parsing, no key lookup. For high-throughput services processing millions of messages, this matters.

### Schema Enforcement

With JSON, your "schema" lives in documentation or a JSON Schema file. With protobuf, the schema is the source of truth:

```protobuf
// user.proto
syntax = "proto3";

message User {
  int32 user_id = 1;
  string name = 2;
  string email = 3;
  bool active = 4;
}
```

Every field has a type, a name, and a field number. The field number (not the name) is what actually gets encoded — this is how you evolve schemas without breaking backwards compatibility.

### TypeScript Code Generation with ts-proto

`ts-proto` is the modern TypeScript protobuf code generator:

```bash
npm install --save-dev ts-proto protoc-gen-ts-proto
```

Create a `.proto` file:

```protobuf
// proto/user.proto
syntax = "proto3";

package user;

option go_package = "./user";

service UserService {
  rpc GetUser (GetUserRequest) returns (User);
  rpc CreateUser (CreateUserRequest) returns (User);
  rpc ListUsers (ListUsersRequest) returns (stream User);
  rpc WatchUsers (google.protobuf.Empty) returns (stream User);
}

message User {
  int32 id = 1;
  string name = 2;
  string email = 3;
  bool active = 4;
  int64 created_at = 5;
}

message GetUserRequest {
  int32 id = 1;
}

message CreateUserRequest {
  string name = 1;
  string email = 2;
}

message ListUsersRequest {
  int32 page = 1;
  int32 page_size = 2;
}
```

Generate TypeScript types:

```bash
protoc \
  --plugin=./node_modules/.bin/protoc-gen-ts_proto \
  --ts_proto_out=./src/generated \
  --ts_proto_opt=esModuleInterop=true,outputServices=grpc-js \
  --proto_path=./proto \
  ./proto/user.proto
```

This generates:
- TypeScript interfaces for all messages
- Client and server class stubs
- Serialization/deserialization code

## gRPC vs REST vs GraphQL vs tRPC

### When to Choose Each

| Scenario | Best Choice | Reason |
|---|---|---|
| Internal microservices | gRPC | Binary efficiency, streaming, codegen |
| Public API, broad client support | REST | Universal support, familiar tooling |
| Complex queries, flexible data fetching | GraphQL | Query flexibility, self-documenting |
| Full-stack TypeScript monorepo | tRPC | Zero overhead, end-to-end types |
| Real-time data streaming | gRPC | Native streaming support |
| Mobile apps on poor connections | gRPC | Smaller payloads |

### The Honest Comparison

**REST**: Simple, universal, cacheable. Any tool can call it. No codegen required. But no streaming, no type enforcement across the wire.

**GraphQL**: Solves over-fetching/under-fetching brilliantly. Complex to implement correctly (N+1 problems, authorization). Less efficient than gRPC for high-throughput.

**tRPC**: Perfect for TypeScript monorepos where client and server share code. Zero runtime overhead. Doesn't work across language boundaries.

**gRPC**: Best for polyglot microservice communication. Requires codegen setup. Browser support requires gRPC-Web or Connect.

## Setting Up a gRPC Server in Node.js

```bash
npm install @grpc/grpc-js @grpc/proto-loader
```

```typescript
// src/server.ts
import * as grpc from '@grpc/grpc-js';
import * as protoLoader from '@grpc/proto-loader';
import path from 'path';

const PROTO_PATH = path.join(__dirname, '../proto/user.proto');

const packageDefinition = protoLoader.loadSync(PROTO_PATH, {
  keepCase: true,
  longs: String,
  enums: String,
  defaults: true,
  oneofs: true,
});

const proto = grpc.loadPackageDefinition(packageDefinition) as any;

// In-memory "database" for the example
const users = new Map([
  [1, { id: 1, name: 'Alice', email: 'alice@example.com', active: true }],
  [2, { id: 2, name: 'Bob', email: 'bob@example.com', active: true }],
]);

const server = new grpc.Server();

server.addService(proto.user.UserService.service, {
  getUser(call: any, callback: any) {
    const user = users.get(call.request.id);
    if (!user) {
      callback({ code: grpc.status.NOT_FOUND, message: 'User not found' });
      return;
    }
    callback(null, user);
  },

  createUser(call: any, callback: any) {
    const id = users.size + 1;
    const user = { id, ...call.request, active: true };
    users.set(id, user);
    callback(null, user);
  },

  // Server-side streaming
  listUsers(call: any) {
    for (const user of users.values()) {
      call.write(user);
    }
    call.end();
  },
});

server.bindAsync('0.0.0.0:50051', grpc.ServerCredentials.createInsecure(), (err) => {
  if (err) throw err;
  console.log('gRPC server running on port 50051');
});
```

## gRPC Client

```typescript
// src/client.ts
import * as grpc from '@grpc/grpc-js';
import * as protoLoader from '@grpc/proto-loader';

const packageDefinition = protoLoader.loadSync('./proto/user.proto');
const proto = grpc.loadPackageDefinition(packageDefinition) as any;

const client = new proto.user.UserService(
  'localhost:50051',
  grpc.credentials.createInsecure()
);

// Unary call
client.getUser({ id: 1 }, (err: any, response: any) => {
  console.log('User:', response);
});

// Server-side streaming
const stream = client.listUsers({ page: 1, page_size: 10 });
stream.on('data', (user: any) => console.log('Received user:', user));
stream.on('end', () => console.log('Stream ended'));
stream.on('error', (err: any) => console.error('Stream error:', err));
```

## Buf CLI: Modern Protobuf Workflow

[Buf](https://buf.build/) has replaced hand-written protoc scripts as the standard protobuf workflow tool.

```bash
npm install -g @bufbuild/buf
```

Initialize a Buf project:

```yaml
# buf.yaml
version: v2
modules:
  - path: proto
    name: buf.build/yourorg/yourapi
lint:
  use:
    - STANDARD
breaking:
  use:
    - FILE
```

Configure code generation:

```yaml
# buf.gen.yaml
version: v2
inputs:
  - directory: proto
plugins:
  - remote: buf.build/bufbuild/es
    out: src/generated
    opt: target=ts
  - remote: buf.build/grpc/node
    out: src/generated
    opt: grpc_js
```

Run generation:

```bash
buf generate  # Generate TypeScript from all .proto files
buf lint      # Check for API design issues
buf breaking  # Check for breaking changes vs last version
buf push      # Publish to Buf Schema Registry
```

### Buf vs protoc

| Feature | protoc | Buf CLI |
|---|---|---|
| Plugin management | Manual install | Automatic |
| Linting | None | Built-in |
| Breaking change detection | None | Built-in |
| Schema registry | None | buf.build registry |
| Remote plugins | No | Yes |
| Learning curve | High | Low |

## gRPC-Web and Connect Protocol

Standard gRPC uses HTTP/2 with custom framing that browsers don't support natively. Two solutions exist:

### gRPC-Web (Envoy Proxy Required)

Traditional gRPC-Web requires an Envoy proxy to translate between browser HTTP/1.1 and gRPC's HTTP/2:

```yaml
# envoy.yaml (simplified)
static_resources:
  listeners:
  - address:
      socket_address:
        address: 0.0.0.0
        port_value: 8080
    filter_chains:
    - filters:
      - name: envoy.filters.network.http_connection_manager
        typed_config:
          "@type": type.googleapis.com/envoy.extensions.filters.network.http_connection_manager.v3.HttpConnectionManager
          codec_type: auto
          route_config:
            virtual_hosts:
            - domains: ["*"]
              routes:
              - route:
                  cluster: grpc_service
          http_filters:
          - name: envoy.filters.http.grpc_web
          - name: envoy.filters.http.cors
          - name: envoy.filters.http.router
```

### Connect Protocol (Recommended)

[Connect](https://connectrpc.com/) by Buf is a better alternative — it supports gRPC, gRPC-Web, and its own Connect protocol, all from the same server, with no proxy required:

```bash
npm install @connectrpc/connect @connectrpc/connect-node @connectrpc/connect-web
```

```typescript
// Generated by buf with connect plugins
import { createClient } from '@connectrpc/connect';
import { createConnectTransport } from '@connectrpc/connect-web';
import { UserService } from './generated/user_connect';

const transport = createConnectTransport({
  baseUrl: 'https://api.example.com',
});

const client = createClient(UserService, transport);

// Works in browsers without any proxy
const user = await client.getUser({ id: 1 });
console.log(user.name);

// Browser-native streaming
for await (const user of client.listUsers({ page: 1, pageSize: 10 })) {
  console.log(user.name);
}
```

## Streaming Patterns

gRPC supports four streaming modes:

### Server-Side Streaming

```typescript
// Real-time log streaming
async function streamLogs(serverId: string) {
  const stream = client.streamLogs({ serverId });

  for await (const log of stream) {
    console.log(`[${log.level}] ${log.message}`);
  }
}
```

### Client-Side Streaming

```typescript
// Bulk data upload
async function uploadMetrics(metrics: Metric[]) {
  const stream = await client.uploadMetrics();

  for (const metric of metrics) {
    await stream.send(metric);
  }

  const result = await stream.closeAndRecv();
  console.log(`Uploaded ${result.count} metrics`);
}
```

### Bidirectional Streaming

```typescript
// Chat application
async function startChat(userId: string) {
  const stream = client.chat();

  // Send messages
  readline.on('line', async (line) => {
    await stream.send({ userId, message: line });
  });

  // Receive messages
  for await (const message of stream) {
    console.log(`${message.fromUserId}: ${message.text}`);
  }
}
```

```
┌──────────────┐     gRPC      ┌──────────────┐
│  API Gateway │ ─────────────▶│ User Service │
│  (REST/GQL)  │               │   :50051     │
└──────────────┘               └──────────────┘
       │                              │

       ▼                              ▼
┌──────────────┐              ┌──────────────┐
│ Order Service│              │ Auth Service │

└──────────────┘              └──────────────┘
```

```typescript
// api-gateway/src/routes/orders.ts
import { createClient } from '@connectrpc/connect';
import { OrderService } from '../generated/order_connect';
import { UserService } from '../generated/user_connect';

export async function getOrderWithUser(orderId: number) {
  const [order, user] = await Promise.all([
    orderClient.getOrder({ id: orderId }),
    userClient.getUser({ id: order.userId }), // parallel fetch
  ]);

  return { ...order, user };
}
```

## Key Takeaways

- **Choose gRPC** for internal microservice communication, streaming, or polyglot environments
- **Use Buf CLI** instead of raw protoc for a much better developer experience
- **Use Connect protocol** for browser clients — avoids the Envoy proxy requirement
- **ts-proto** generates idiomatic TypeScript from .proto files
- **grpcurl** is your debugging companion, equivalent to curl for REST
- **Schema evolution** is safe as long as you don't change field numbers — Buf's breaking change detector catches violations

The JavaScript gRPC ecosystem has matured significantly. The combination of Buf CLI, ts-proto, and Connect makes what used to be a painful setup into a tractable workflow.
