"Never trust, always verify" — zero trust's core principle sounds obvious but its implications are radical. In a traditional perimeter model, everything inside the corporate network is implicitly trusted. Zero trust rejects that assumption entirely: every request must be authenticated and authorized, regardless of where it originates. Even a service calling another service within your own Kubernetes cluster must prove its identity.

This guide explains zero trust for developers — not the marketing buzzword version, but the practical patterns you implement in code and infrastructure.

## Why Zero Trust? The Lateral Movement Problem

Traditional security builds a castle-and-moat model: strong perimeter, soft interior. Once an attacker breaches the perimeter (phished credential, compromised dependency, SSRF vulnerability), they can move laterally through the network with few obstacles. They pivot from a compromised web server to an internal database to an S3 bucket to your backup systems — all without triggering alerts because internal traffic is trusted.

Zero trust eliminates implicit trust at every hop:
- **Every service** must authenticate to every other service
- **Every user** must authenticate, even on the corporate network
- **Every device** must meet a security baseline to receive access
- **Access is scoped** to the minimum necessary (least privilege)

## The Three Pillars for Developers

### Pillar 1: Mutual TLS (mTLS) — Service Identity

Regular TLS verifies the server's identity (your browser trusts the server's certificate). **Mutual TLS** adds the other direction: the server also verifies the client's certificate. This gives every service in your system a cryptographic identity.

Without mTLS: Service A calls Service B's HTTP endpoint. Anyone on the network can call Service B. A compromised Service A can impersonate any service.

With mTLS: Service A presents its certificate. Service B verifies it against the trust root. Service B knows, cryptographically, that it's talking to Service A — not an attacker pretending to be Service A.

**Implementing mTLS directly (Go example):**

```go
// Server: require client certificate
tlsConfig := &tls.Config{
    ClientAuth: tls.RequireAndVerifyClientCert,
    ClientCAs:  certPool, // trust root CA
}
server := &http.Server{
    TLSConfig: tlsConfig,
}

// Client: present certificate
cert, _ := tls.LoadX509KeyPair("service-a.crt", "service-a.key")
transport := &http.Transport{
    TLSClientConfig: &tls.Config{
        Certificates: []tls.Certificate{cert},
        RootCAs:      certPool,
    },
}
client := &http.Client{Transport: transport}
```

Managing certificates manually at scale is complex. This is why service meshes exist.

### Pillar 2: Service Mesh — mTLS at Scale

A service mesh (Istio, Linkerd, Cilium) implements mTLS transparently via sidecar proxies or eBPF. Your application code doesn't change — the mesh injects and rotates certificates automatically, enforces mTLS on all service-to-service traffic, and provides audit logs of every connection.

**Istio: enable strict mTLS cluster-wide**

```yaml
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: istio-system  # applies cluster-wide
spec:
  mtls:
    mode: STRICT  # reject any non-mTLS traffic
```

With this policy, every pod-to-pod connection in the cluster requires mTLS. Istio's control plane (Istiod) acts as the CA, issues short-lived SPIFFE certificates to each workload, and rotates them automatically before expiry.

**Authorization policies: only allow known callers**

```yaml
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: order-service-policy
  namespace: production
spec:
  selector:
    matchLabels:
      app: order-service
  rules:
  - from:
    - source:
        principals:
          - "cluster.local/ns/production/sa/checkout-service"
          - "cluster.local/ns/production/sa/admin-service"
    to:
    - operation:
        methods: ["POST", "GET"]
        paths: ["/api/orders*"]
```

Only the `checkout-service` and `admin-service` ServiceAccounts can call `order-service`. Any other service — even within the same cluster — gets denied.

### Pillar 3: Identity-Based Access Control

Zero trust shifts from network-based access (IP allowlists, VPN) to identity-based access (who/what is making this request?). Three layers matter:

**Workload Identity (SPIFFE/SPIRE)**

Every workload gets a SPIFFE ID: `spiffe://example.com/ns/production/sa/order-service`. This is the cryptographic identity used in mTLS certificates. It's bound to the Kubernetes ServiceAccount, not an IP or hostname.

**User Identity (Zero Trust Network Access)**

Tools like Cloudflare Access, Tailscale, or BeyondCorp replace VPN. Users authenticate with SSO (OIDC/SAML) and access is granted per-application, not per-network segment. A compromised VPN credential no longer gives access to "everything internal."

```yaml
# Cloudflare Access policy example
policies:
  - name: "Internal Dashboard Access"
    decision: allow
    include:
      - email_domain: "yourcompany.com"
      - group: "engineering"
    require:
      - mfa: true
      - device_posture:
          check:
            type: disk_encryption
            enabled: true
```

**Least Privilege with Short-Lived Credentials**

Pair workload identity with a secrets manager like [HashiCorp Vault](/tools/hashicorp-vault) or AWS IAM roles:

```python
# Kubernetes pod assumes IAM role via IRSA
# No static AWS credentials stored anywhere
import boto3

# boto3 auto-fetches credentials from the pod's projected token
s3 = boto3.client('s3')
# The IAM role has access only to the specific bucket this service needs
```

## Practical Implementation Path

Zero trust is a journey, not a switch. Here's a realistic rollout path:

**Phase 1 — Stop the bleeding (1-2 weeks)**
- Enable [Snyk](/tools/snyk) or a similar SCA scanner in CI/CD
- Migrate secrets from `.env` files to a secrets manager
- Enable cloud provider audit logs (CloudTrail, Cloud Audit Logs)

**Phase 2 — Service identity (1-2 months)**
- Deploy a service mesh (Linkerd is simpler than Istio for initial adoption)
- Enable mTLS in permissive mode (observe, don't block yet)
- Audit service-to-service traffic patterns

**Phase 3 — Enforce and restrict (ongoing)**
- Switch mTLS to STRICT mode
- Implement AuthorizationPolicies for sensitive services
- Replace VPN with zero trust network access for developer access
- Implement device posture checks

## Common Pitfalls

**Certificate rotation gaps:** Short-lived certs (24h) are great security but require your applications to reload credentials without restarting. Test this explicitly.

**Overly broad SPIFFE IDs:** A ServiceAccount with permissions to call any service defeats the purpose. Map each service to a dedicated ServiceAccount with minimal permissions.

**Ignoring egress:** Most zero trust implementations focus on east-west (service-to-service) traffic. Don't forget egress — outbound calls to the internet should be restricted to allowlisted destinations and proxied through an egress gateway.

**mTLS for external clients:** mTLS between internal services is great. External clients (browsers, mobile apps) can't easily present client certificates. Use OIDC tokens (short-lived JWTs) for external authentication instead, and validate them at your ingress/API gateway.

Zero trust is not a product you buy — it's an architecture you build incrementally. Start with identity, layer in mTLS, enforce least privilege, and audit everything. Each step meaningfully reduces your blast radius.
