# Vite vs Webpack vs esbuild: Build Tool Benchmark & Guide

The JavaScript build tool ecosystem has been overhauled. Webpack, which dominated for nearly a decade, is now being replaced by faster alternatives. Vite has become the default for modern frontend development, while esbuild and Rollup serve specific niches. Here's what you need to know.

---

## At a Glance

| | Webpack 5 | Vite | esbuild |
|---|---|---|---|
| Built with | JavaScript | JavaScript + Rollup/esbuild | Go |
| Dev server startup | ~5-30s | <1s | N/A (library) |
| HMR speed | Slow (full module graph) | Fast (native ESM) | N/A |
| Production bundle | Very mature | Rollup (excellent tree-shaking) | Fast but less configurable |
| Config complexity | High | Low | Minimal |
| Ecosystem | Massive | Growing fast | Small |
| Use case | Complex enterprise apps | Modern web apps | Libraries, CLIs, internal tools |

## Webpack 5: The Veteran

Webpack is the most battle-tested bundler. It handles every edge case, has a plugin for everything, and powers some of the most complex front-end deployments in the world (Next.js before 13 used Webpack as its primary bundler).

### The trade-off: configuration complexity

```javascript
// webpack.config.js — even a basic config is verbose
const path = require('path');
const HtmlWebpackPlugin = require('html-webpack-plugin');
const MiniCssExtractPlugin = require('mini-css-extract-plugin');

module.exports = {
  entry: './src/index.ts',
  output: {
    path: path.resolve(__dirname, 'dist'),
    filename: '[name].[contenthash].js',
    clean: true,
  },
  module: {
    rules: [
      {
        test: /\.tsx?$/,
        use: 'babel-loader',
        exclude: /node_modules/,
      },
      {
        test: /\.css$/,
        use: [MiniCssExtractPlugin.loader, 'css-loader', 'postcss-loader'],
      },
    ],
  },
  plugins: [
    new HtmlWebpackPlugin({ template: './public/index.html' }),
    new MiniCssExtractPlugin({ filename: '[name].[contenthash].css' }),
  ],
  resolve: { extensions: ['.ts', '.tsx', '.js'] },
};
```

### When Webpack is still the right choice

- **Legacy migration** — upgrading a Webpack 4 codebase to Webpack 5 is safer than switching tools
- **Module Federation** — Webpack 5's Module Federation is the gold standard for micro-frontend architectures
- **Complex custom transforms** — Webpack's loader/plugin system has no equal for truly custom pipelines
- **Next.js apps** — Next.js's Webpack integration is deeply optimized

## Vite: The Modern Default

Vite was created by Vue.js author Evan You to solve the slow dev server problem. Instead of bundling everything on startup, Vite serves files as native ES modules — the browser imports them directly. Only changed files are recompiled.

### Zero-config startup

```bash
npm create vite@latest my-app -- --template react-ts
cd my-app && npm install && npm run dev
# Dev server ready in < 500ms
```

### Vite config (much simpler than Webpack)

```typescript
// vite.config.ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react()],
  build: {
    target: 'esnext',
    minify: 'esbuild',  // uses esbuild for minification
    rollupOptions: {
      output: {
        manualChunks: {
          vendor: ['react', 'react-dom'],
        },
      },
    },
  },
  server: {
    proxy: {
      '/api': 'http://localhost:3000',
    },
  },
});
```

### Why Vite's dev server is fast

```
Traditional bundler dev server:
Request → Bundle entire app → Serve bundle (slow on large apps)

Vite dev server:
Request → Serve the specific file as ESM → Browser requests imports directly
```

No bundling on start = instant dev server. HMR only invalidates the changed module.

### Production build with Rollup

For production, Vite uses Rollup (not esbuild) to generate optimally tree-shaken, chunked output:

```bash
vite build
# Output: dist/
# └── assets/
#     ├── index-abc123.js    (your app code)
#     ├── vendor-def456.js   (React, etc.)
#     └── index-ghi789.css
```

## esbuild: The Speed Demon

esbuild is written in Go and is 10-100x faster than JavaScript-based bundlers. It's primarily used as a **library** inside other tools (Vite uses it for transforms, Webpack has an esbuild loader), not as a standalone dev workflow.

### esbuild raw speed

```bash
# Bundle a 1,000-module React app
esbuild --bundle src/index.tsx --outfile=dist/bundle.js  # ~40ms
webpack (JS)                                              # ~3000ms
```

### Using esbuild directly

```javascript
import * as esbuild from 'esbuild';

await esbuild.build({
  entryPoints: ['src/index.ts'],
  bundle: true,
  minify: true,
  splitting: true,
  format: 'esm',
  outdir: 'dist',
  external: ['react', 'react-dom'],  // don't bundle, import from CDN
  define: {
    'process.env.NODE_ENV': '"production"',
  },
});
```

### esbuild's limitations

- No CSS Modules support (needs plugins)
- Less sophisticated code splitting than Rollup
- No HMR dev server built-in
- TypeScript type checking is intentionally omitted (use `tsc --noEmit` separately)

### Best use cases for esbuild

- **Node.js libraries** — bundle TypeScript to CJS/ESM with zero overhead
- **CLI tools** — compile to a single file instantly
- **Internal build scripts** — fast compilation without a full bundler setup
- **Vite/Vitest underlying engine** — you're probably already using it

```bash
# Build a TypeScript CLI to a single binary-friendly JS file
esbuild src/cli.ts --bundle --platform=node --outfile=dist/cli.js
```

## Choosing the Right Tool

**Use Vite for:**
- New React, Vue, Svelte, or Solid apps
- Projects where dev experience matters
- When you're starting from scratch with no legacy constraints

**Use Webpack for:**
- Existing Webpack projects (upgrade 4→5, don't switch tools)
- Micro-frontend architectures with Module Federation
- Complex custom build pipelines with specific loaders

**Use esbuild for:**
- Node.js library bundling
- TypeScript compilation to JS (faster than tsc)
- Build scripts, CLIs, and internal tools
- When you embed it in a custom toolchain

**Avoid mixing tools** — picking Vite for frontend and esbuild for backend means two different mental models. Bun bundles both scenarios with a unified `bun build` command.
