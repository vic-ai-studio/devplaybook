# Backstage vs Port vs Cortex: IDP Comparison Guide 2026

When it comes to building an Internal Developer Platform (IDP), three tools dominate the conversation: **Backstage** (open-source by Spotify), **Port** (modern SaaS), and **Cortex** (engineering intelligence SaaS). Each takes a different approach to the same problem: helping developers find information, provision infrastructure, and operate services efficiently.

This guide compares all three tools on the dimensions that matter most for your decision.

## Overview

### Backstage

Created by Spotify in 2020 and donated to the CNCF (Cloud Native Computing Foundation), Backstage is the most widely adopted developer portal framework in the world. It's open-source, self-hosted, and extensible through a plugin system.

Backstage reached CNCF graduated status in 2023 — the same tier as Kubernetes and Prometheus — validating its maturity and ecosystem. Major adopters include Spotify, American Airlines, Expedia, Netflix (Vibes), Box, and hundreds of other organizations.

### Port

Port launched in 2022 as a modern, API-first alternative to Backstage. It's a SaaS product with a flexible data model that lets you define custom entities, relationships, and self-service actions without writing code.

Port has grown rapidly, positioning itself as the "next-generation IDP" with better time to value and more modern UX than Backstage.

### Cortex

Cortex (cortex.io) focuses specifically on service catalog, scorecards, and engineering intelligence. It's narrower than Backstage or Port but excels at its specific use case: giving organizations visibility into service ownership, health, and compliance.

Cortex is often used alongside a broader portal solution or as a standalone tool for engineering intelligence.

## Feature Comparison

### Service Catalog

**Backstage**: The software catalog is Backstage's core feature. Entities are defined in `catalog-info.yaml` files in your repos. The catalog supports `Component`, `API`, `Resource`, `System`, `Domain`, `Group`, and `User` entity types. Highly flexible with custom annotations.

**Port**: Port's catalog (called "Software Catalog") uses a flexible entity model where you define your own "blueprints" (schemas) for entities. This is more flexible than Backstage's fixed entity types. You can model services, Kubernetes clusters, databases, features, or anything else with custom properties.

**Port example blueprint:**
```json
{
  "identifier": "service",
  "title": "Service",
  "schema": {
    "properties": {
      "language": { "type": "string" },
      "health_status": { "type": "string" },
      "deployment_frequency": { "type": "number" }
    },
    "required": ["language"]
  }
}
```

**Cortex**: Cortex catalog is focused on services and APIs. Less flexible than Port but cleaner for the specific use case of service ownership.

**Winner for catalog flexibility**: Port > Backstage > Cortex

### Software Templates / Scaffolding

**Backstage**: Backstage's Scaffolder is a powerful templating engine that creates repos, configures CI/CD, and registers services in the catalog. Uses Nunjucks templating syntax. Supports complex multi-step workflows.

```yaml
# backstage scaffolder template
steps:
  - id: create-repo
    action: publish:github
    input:
      repoUrl: 'github.com?repo={{ parameters.name }}&owner=myorg'
  - id: setup-ci
    action: github:actions:dispatch
    input:
      workflowId: setup-service.yml
  - id: register
    action: catalog:register
```

**Port**: Port's "self-service actions" can trigger scaffolding workflows via webhooks, GitHub Actions, or their backend. Less opinionated than Backstage's scaffolder but more flexible for complex multi-system workflows.

**Cortex**: No built-in scaffolding. Not a core use case.

**Winner for scaffolding**: Backstage > Port > Cortex

### Self-Service Actions

**Backstage**: Via the Scaffolder or custom backend plugins. Requires writing TypeScript/Node.js code to implement custom actions.

**Port**: Port's self-service hub supports complex action workflows with conditional steps, approvals, and integrations. Can trigger GitHub Actions, webhooks, Terraform runs, or custom scripts. No code required for many common actions.

**Cortex**: Limited self-service actions. Primarily read-only catalog with some automation.

**Winner for self-service actions**: Port > Backstage > Cortex

### Scorecards and Standards

**Backstage**: Backstage doesn't have built-in scorecards. You need the TechInsights plugin (community-maintained) to add scorecard functionality.

**Port**: Port has "scorecards" as a built-in feature. You define standards (e.g., "every service must have an owner, documentation, and monitoring") and Port automatically evaluates compliance across your catalog.

**Cortex**: Scorecards are Cortex's flagship feature. Define custom standards (documentation, on-call coverage, SLO configuration, dependency freshness) and track compliance trends over time. Excellent UI for visualizing organizational compliance.

**Winner for scorecards**: Cortex > Port > Backstage

### Documentation

**Backstage**: TechDocs is a first-class Backstage feature. Uses MkDocs under the hood, colocated in service repos. Docs are automatically built and published to the portal. Excellent for keeping docs close to code.

**Port**: Port has documentation capabilities but doesn't have a native docs-as-code system like TechDocs. Can display links to docs and embed external content.

**Cortex**: Documentation management is not a primary feature. Can display doc links but no native hosting.

**Winner for documentation**: Backstage > Port > Cortex

### Kubernetes Integration

**Backstage**: Excellent Kubernetes plugin that displays pod status, resource usage, events, and deployment history directly in the service catalog. The `@backstage/plugin-kubernetes` plugin is one of the most popular in the ecosystem.

**Port**: Port has Kubernetes integrations via their platform agent, which can sync cluster state to your catalog. Good but less polished than Backstage's dedicated plugin.

**Cortex**: Basic Kubernetes integration, not a focus area.

**Winner for Kubernetes**: Backstage > Port > Cortex

## Setup and Time-to-Value

### Backstage

**Setup time**: 1–4 weeks to get a basic portal running. 1–3 months to get full value with customization, templates, and catalog population.

**Deployment requirements**: You host it yourself (Docker, Kubernetes, or cloud-native). Requires PostgreSQL. Ongoing DevOps maintenance.

**Technical requirements**: TypeScript/React knowledge for plugin development. Node.js for backend.

**First-day experience**:
```bash
npx @backstage/create-app@latest
cd my-backstage-app
yarn dev
# Portal running at localhost:3000 — but catalog is empty
```

Initial setup is easy. Making it useful takes months.

### Port

**Setup time**: Hours to days for initial setup. 1–4 weeks to fully populate catalog and configure self-service actions.

**Deployment requirements**: SaaS — no infrastructure to manage. Install integrations via their platform agent.

**Technical requirements**: No code required for basic setup. API/scripting knowledge for advanced customization.

**First-day experience**:
- Sign up at port.getport.io
- Install the GitHub integration
- Port automatically discovers your repositories
- Begin configuring blueprints and scorecards

Port's time-to-value is significantly faster than Backstage.

### Cortex

**Setup time**: 1–3 days for initial catalog population with Git integrations.

**Deployment requirements**: SaaS.

**Technical requirements**: Minimal — configure integrations via UI.

**First-day experience**: Similar to Port — SaaS signup + integration connectors = catalog visible quickly.

## Pricing

### Backstage

**Cost**: Free (open-source)

**Actual cost**: While the software is free, you pay in engineering time:
- Initial setup: 2–8 weeks of platform engineer time
- Ongoing maintenance: 0.5–1 FTE per year
- Plugin development for custom needs: additional time

For a small team, Backstage can be more expensive than SaaS alternatives when you factor in engineering cost.

### Port

**Pricing**: Per-seat SaaS (approximately $35–60/user/month for paid plans)

**Free tier**: Available with limited features — good for evaluation.

**Enterprise**: Custom pricing for large organizations.

At 50 users: ~$1,750–3,000/month. At 200 users: ~$7,000–12,000/month.

### Cortex

**Pricing**: Per-seat SaaS (approximately $20–40/user/month, varies by tier)

**Free tier**: Limited features, good for small teams to evaluate.

**Enterprise**: Custom pricing.

Cortex is generally priced lower than Port, reflecting its narrower feature set.

## Migration Considerations

### Backstage to Port

Many organizations start with Backstage and move to Port (or run both) as they want more self-service capabilities without the maintenance burden. The service catalog can be migrated via Port's GitHub integration (it reads the same `catalog-info.yaml` files).

### Port to Backstage

Less common (Port usually wins on time to value), but organizations with strict self-hosting requirements sometimes migrate from Port to Backstage.

### Using Both

Some large organizations use both: Backstage for the developer-facing portal (TechDocs, templates, Kubernetes visualization) and Cortex for engineering intelligence and scorecard tracking. This is a valid but complex approach.

## Decision Framework

### Choose Backstage if:
- You have a dedicated platform engineering team (2+ engineers)
- You need deep customization and can invest months in setup
- Self-hosting is a requirement (compliance, security)
- Kubernetes integration is critical
- You need colocated documentation (TechDocs)
- Budget is constrained (you have engineering time but not SaaS budget)

### Choose Port if:
- You need a portal running this quarter
- Your team is small (1 platform engineer, or none dedicated)
- You want no infrastructure to maintain
- Flexibility in data modeling is important
- You need strong self-service actions without custom code
- Budget for SaaS is available

### Choose Cortex if:
- Service ownership accountability is your primary pain point
- You want engineering health scorecards
- You already have another portal solution or don't need templates/scaffolding
- Compliance tracking is critical

### Hybrid Approach:
- **Backstage + Cortex**: Use Backstage for portal/templates/docs, Cortex for scorecards
- **Port + Cortex**: Port for catalog/self-service, Cortex for engineering intelligence
- **Start with Port, migrate to Backstage**: Get value fast, add customization later

## Conclusion

There's no single best tool — it depends on your team's size, technical resources, timeline, and priorities.

For most organizations in 2026:
- **Start with Port** if you want a portal running quickly with minimal engineering investment
- **Use Backstage** if you need deep customization, TechDocs, and are willing to invest the engineering time
- **Add Cortex** if scorecard-based accountability and engineering health tracking is a priority

The most important thing is to ship something. A functional portal built in two weeks with Port delivers more value than a perfectly designed Backstage instance that's still being planned six months later.
