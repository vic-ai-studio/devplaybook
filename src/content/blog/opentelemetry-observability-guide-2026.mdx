Observability is no longer optional for production systems. When a user reports slowness, you need to answer: *which service is slow, which database query is the culprit, and which deploy introduced it?* OpenTelemetry (OTel) is the open standard that makes answering these questions possible — across languages, frameworks, and vendors.

In 2026, OpenTelemetry has reached stable 1.x releases for the three pillars of observability: **traces**, **metrics**, and **logs**. This guide walks through instrumentation, collection, and visualization with a practical production setup.

## What is OpenTelemetry?

OpenTelemetry is a CNCF project that standardizes how you instrument applications for observability. Before OTel, every vendor (Datadog, New Relic, Jaeger, Zipkin) had its own SDK. You'd instrument once for Jaeger, then re-instrument everything when you switched to Datadog. OTel solves this with a vendor-neutral API and SDK — instrument once, route to any backend.

The three pillars OTel covers:

- **Traces:** End-to-end request flows across services (distributed tracing)
- **Metrics:** Numerical time-series data (latency, error rates, queue depth)
- **Logs:** Structured event records with trace correlation

## Architecture: The OTel Collector

The recommended production setup routes all telemetry through the **OTel Collector** — a vendor-agnostic proxy that receives, processes, and exports data to multiple backends simultaneously.

```
App (OTel SDK) → OTel Collector → Jaeger (traces)
                               → Prometheus (metrics)
                               → Loki (logs)
                               → Datadog (any SaaS)
```

```yaml
# otel-collector-config.yaml
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318

processors:
  batch:
    timeout: 1s
    send_batch_size: 1024

exporters:
  jaeger:
    endpoint: jaeger:14250
    tls:
      insecure: true
  prometheus:
    endpoint: "0.0.0.0:8889"
  loki:
    endpoint: http://loki:3100/loki/api/v1/push

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch]
      exporters: [jaeger]
    metrics:
      receivers: [otlp]
      processors: [batch]
      exporters: [prometheus]
    logs:
      receivers: [otlp]
      processors: [batch]
      exporters: [loki]
```

## Instrumenting Your Application

### Auto-Instrumentation (Zero Code Changes)

For common frameworks, OTel provides auto-instrumentation that injects spans without touching your code:

```bash
# Python — auto-instrument Flask/FastAPI/Django
pip install opentelemetry-distro opentelemetry-exporter-otlp
opentelemetry-bootstrap -a install

# Run with auto-instrumentation
opentelemetry-instrument \
  --traces_exporter otlp \
  --metrics_exporter otlp \
  --logs_exporter otlp \
  --exporter_otlp_endpoint http://otel-collector:4317 \
  python app.py
```

```bash
# Node.js — auto-instrument Express/Fastify/NestJS
npm install @opentelemetry/auto-instrumentations-node @opentelemetry/sdk-node

# Add to startup
node --require @opentelemetry/auto-instrumentations-node/register server.js
```

Auto-instrumentation captures: HTTP request spans, database queries, Redis calls, outbound HTTP calls, and message queue operations — all with zero code changes.

### Manual Instrumentation for Business Logic

Auto-instrumentation covers infrastructure. For business-critical code paths, add manual spans:

```python
from opentelemetry import trace
from opentelemetry.trace import Status, StatusCode

tracer = trace.get_tracer("order-service")

async def process_order(order_id: str):
    with tracer.start_as_current_span("process_order") as span:
        span.set_attribute("order.id", order_id)
        span.set_attribute("order.source", "web")

        try:
            order = await db.fetch_order(order_id)
            span.set_attribute("order.amount", order.total_cents)

            with tracer.start_as_current_span("validate_inventory"):
                await inventory.check(order.items)

            with tracer.start_as_current_span("charge_payment"):
                charge = await payments.charge(order)
                span.set_attribute("payment.charge_id", charge.id)

            span.set_status(Status(StatusCode.OK))
        except InsufficientInventoryError as e:
            span.record_exception(e)
            span.set_status(Status(StatusCode.ERROR, str(e)))
            raise
```

### Custom Metrics with OTel SDK

```python
from opentelemetry import metrics

meter = metrics.get_meter("order-service")

# Counter: total orders processed
orders_counter = meter.create_counter(
    "orders.processed",
    description="Total number of orders processed",
)

# Histogram: order processing latency
latency_histogram = meter.create_histogram(
    "orders.processing_duration",
    unit="ms",
    description="Order processing latency distribution",
)

async def process_order(order_id: str):
    start = time.monotonic()
    try:
        await _process(order_id)
        orders_counter.add(1, {"status": "success", "source": "web"})
    finally:
        duration_ms = (time.monotonic() - start) * 1000
        latency_histogram.record(duration_ms)
```

## Visualization: Grafana + Jaeger + Prometheus

### Traces in Jaeger / Grafana Tempo

[Grafana](/tools/grafana) Tempo is the recommended trace backend for the LGTM stack. Jaeger works well standalone. Both let you:

- Search traces by service, operation, duration, error status
- View waterfall diagrams showing span relationships and timing
- Jump from a slow span to the exact database query
- Correlate a trace ID from logs directly to the trace

### Metrics in Prometheus / Grafana

Auto-instrumentation exports RED metrics (Rate, Errors, Duration) automatically:

```promql
# Request rate per service
sum(rate(http_server_duration_count[5m])) by (service_name)

# Error rate
sum(rate(http_server_duration_count{http_status_code=~"5.."}[5m])) by (service_name)
  /
sum(rate(http_server_duration_count[5m])) by (service_name)

# P99 latency
histogram_quantile(0.99,
  sum(rate(http_server_duration_bucket[5m])) by (le, service_name)
)
```

### Correlating Logs and Traces

The killer feature of OTel: every log line automatically includes the current `trace_id` and `span_id`. In Grafana, click a log line → jump directly to the trace.

```python
import structlog
from opentelemetry import trace

def add_trace_context(logger, method, event_dict):
    current_span = trace.get_current_span()
    if current_span.is_recording():
        ctx = current_span.get_span_context()
        event_dict["trace_id"] = format(ctx.trace_id, "032x")
        event_dict["span_id"] = format(ctx.span_id, "016x")
    return event_dict

log = structlog.wrap_logger(
    structlog.get_logger(),
    processors=[add_trace_context, structlog.processors.JSONRenderer()]
)
```

## Production Tips

**1. Sample intelligently.** Tracing every request at 100% is expensive. Use tail-based sampling to keep 100% of error traces and slow requests (>1s), but sample only 1% of healthy fast requests.

```yaml
# OTel Collector: tail sampling processor
processors:
  tail_sampling:
    decision_wait: 10s
    policies:
      - name: errors-policy
        type: status_code
        status_code: {status_codes: [ERROR]}
      - name: slow-traces-policy
        type: latency
        latency: {threshold_ms: 1000}
      - name: probabilistic-policy
        type: probabilistic
        probabilistic: {sampling_percentage: 1}
```

**2. Use semantic conventions.** OTel defines standard attribute names (`http.method`, `db.system`, `rpc.service`). Following them means your dashboards and queries work across services without customization.

**3. Set resource attributes at startup.** Service name, version, and environment should be on every telemetry item:

```bash
OTEL_SERVICE_NAME=order-service \
OTEL_SERVICE_VERSION=1.2.3 \
OTEL_DEPLOYMENT_ENVIRONMENT=production \
opentelemetry-instrument python app.py
```

**4. Monitor the collector itself.** The OTel Collector exposes its own metrics. Alert on: dropped spans, export errors, queue depth. A silently failing collector means missing observability data exactly when you need it most.
