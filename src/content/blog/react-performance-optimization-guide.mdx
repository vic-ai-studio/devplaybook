# React Performance Optimization: Profiling, Memoization, and Code Splitting

Most React performance problems aren't framework limitations — they're unnecessary re-renders, unoptimized lists, and under-split bundles. This guide covers profiling-first optimization: measure before you fix.

---

## Step 1: Measure with React DevTools Profiler

Never optimize blindly. Install React DevTools and use the Profiler tab:

```bash
# Add flamegraph visualization
npm install --save-dev @welldone-software/why-did-you-render
```

```typescript
// src/wdyr.ts — add before any React imports in development
import React from 'react';

if (process.env.NODE_ENV === 'development') {
  const whyDidYouRender = require('@welldone-software/why-did-you-render');
  whyDidYouRender(React, {
    trackAllPureComponents: true,
    logOwnerReasons: true,
  });
}
```

```tsx
// Mark specific components to track
ProductList.whyDidYouRender = true;
```

Look for components that re-render with identical props — those are your optimization targets.

## React.memo: Skip Re-renders for Pure Components

`React.memo` prevents a component from re-rendering if its props haven't changed:

```tsx
// Without memo — re-renders whenever parent renders, even if props are the same
function ProductCard({ product, onAddToCart }: Props) {
  console.log('ProductCard render:', product.id);
  return (
    <div>
      <h3>{product.name}</h3>
      <button onClick={() => onAddToCart(product.id)}>Add to Cart</button>
    </div>
  );
}

// With memo — skips re-render if props are shallowly equal
const ProductCard = React.memo(function ProductCard({ product, onAddToCart }: Props) {
  return (
    <div>
      <h3>{product.name}</h3>
      <button onClick={() => onAddToCart(product.id)}>Add to Cart</button>
    </div>
  );
});

// Custom equality function for complex props
const ProductCard = React.memo(
  function ProductCard({ product, filters }: Props) { /* … */ },
  (prevProps, nextProps) => {
    return prevProps.product.id === nextProps.product.id &&
           prevProps.product.price === nextProps.product.price &&
           prevProps.filters.category === nextProps.filters.category;
  }
);
```

**When memo is worth it**: Components that render often, with identical props, and whose render cost is non-trivial (list items, charts, complex forms).

**When NOT to use memo**: Simple components with 1-2 props that always change. The comparison cost can exceed render cost.

## useCallback and useMemo: Stable References

```tsx
function ProductList({ products, userId }: Props) {
  // ❌ New function instance on every render — breaks React.memo on children
  const handleAddToCart = (productId: string) => {
    addToCart(userId, productId);
    trackEvent('add_to_cart', { productId, userId });
  };

  // ✅ Stable reference — only recreated when userId changes
  const handleAddToCart = useCallback((productId: string) => {
    addToCart(userId, productId);
    trackEvent('add_to_cart', { productId, userId });
  }, [userId]);

  // ❌ Expensive computation on every render
  const sortedProducts = products
    .filter(p => p.inStock)
    .sort((a, b) => a.price - b.price);

  // ✅ Only recomputed when products array changes
  const sortedProducts = useMemo(() =>
    products.filter(p => p.inStock).sort((a, b) => a.price - b.price),
    [products]
  );

  return sortedProducts.map(p => (
    <ProductCard key={p.id} product={p} onAddToCart={handleAddToCart} />
  ));
}
```

**The trap**: Overusing `useCallback`/`useMemo` adds overhead. The comparison + closure allocation often costs more than just re-rendering. Use them when:
- The memoized value is genuinely expensive (>1ms)
- The stable reference breaks a downstream optimization (`React.memo`, `useEffect` deps)

## Virtual Lists for Large Data

Never render 10,000 DOM nodes. Use virtualization:

```bash
npm install @tanstack/react-virtual
```

```tsx
import { useVirtualizer } from '@tanstack/react-virtual';

function VirtualProductList({ products }: { products: Product[] }) {
  const parentRef = useRef<HTMLDivElement>(null);

  const virtualizer = useVirtualizer({
    count: products.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 80, // estimated item height in px
    overscan: 5,            // extra items above/below viewport
  });

  return (
    <div
      ref={parentRef}
      style={{ height: '600px', overflow: 'auto' }}
    >
      {/* Total height placeholder for correct scrollbar */}
      <div style={{ height: `${virtualizer.getTotalSize()}px`, position: 'relative' }}>
        {virtualizer.getVirtualItems().map(virtualItem => (
          <div
            key={virtualItem.key}
            style={{
              position: 'absolute',
              top: 0,
              left: 0,
              width: '100%',
              height: `${virtualItem.size}px`,
              transform: `translateY(${virtualItem.start}px)`,
            }}
          >
            <ProductCard product={products[virtualItem.index]} />
          </div>
        ))}
      </div>
    </div>
  );
}
```

For variable-height items:
```tsx
const virtualizer = useVirtualizer({
  count: items.length,
  getScrollElement: () => parentRef.current,
  estimateSize: () => 45,
  measureElement: (el) => el?.getBoundingClientRect().height ?? 45,
});
```

## Code Splitting with React.lazy

Ship only what's needed for the current page:

```tsx
import { lazy, Suspense } from 'react';

// Bundle is NOT included in main chunk — loaded when first rendered
const HeavyChart = lazy(() => import('./HeavyChart'));
const AdminPanel = lazy(() => import('./AdminPanel'));
const PDFViewer = lazy(() => import('./PDFViewer'));

function App() {
  const { isAdmin } = useUser();

  return (
    <Suspense fallback={<ChartSkeleton />}>
      {/* Loaded on-demand */}
      <HeavyChart data={chartData} />

      {/* Only loaded for admins */}
      {isAdmin && (
        <Suspense fallback={<div>Loading admin panel…</div>}>
          <AdminPanel />
        </Suspense>
      )}
    </Suspense>
  );
}
```

Route-level code splitting (Next.js App Router does this automatically):

```tsx
// Each page is automatically code-split in Next.js App Router
// For manual splitting in React Router:
import { createBrowserRouter } from 'react-router-dom';

const router = createBrowserRouter([
  {
    path: '/',
    element: <Layout />,
    children: [
      { index: true, element: <Home /> },
      {
        path: 'dashboard',
        lazy: async () => {
          const { Dashboard } = await import('./pages/Dashboard');
          return { Component: Dashboard };
        },
      },
    ],
  },
]);
```

## Avoid Expensive Context Re-renders

Context re-renders all consumers when value changes. Split contexts:

```tsx
// ❌ One big context — any state change re-renders ALL consumers
const AppContext = createContext<{
  user: User;
  theme: Theme;
  cart: CartItem[];
  settings: Settings;
}>({});

// ✅ Split by update frequency
const UserContext = createContext<User | null>(null);     // Changes rarely
const ThemeContext = createContext<Theme>('light');        // Changes rarely
const CartContext = createContext<CartItem[]>([]);         // Changes often

// Or use context selectors with useSyncExternalStore for granular subscriptions
function useCartItemCount() {
  return useSyncExternalStore(cartStore.subscribe, () => cartStore.getItemCount());
}
```

## The React Compiler (Auto-Memoization)

React Compiler (RC, previously React Forget) automatically memoizes components and hooks:

```bash
npm install babel-plugin-react-compiler
```

```js
// babel.config.js
module.exports = {
  plugins: [
    ['babel-plugin-react-compiler', { target: '18' }]
  ]
};
```

With the compiler enabled, this code is automatically optimized:

```tsx
// Without compiler — you'd need useCallback and useMemo
function Form({ onSubmit }: Props) {
  const [name, setName] = useState('');

  const handleSubmit = () => { // Compiler auto-memoizes this
    onSubmit(name);
  };

  const isValid = name.length > 2; // Compiler auto-memoizes this

  return (
    <input value={name} onChange={e => setName(e.target.value)} />
    <button disabled={!isValid} onClick={handleSubmit}>Submit</button>
  );
}
```

The compiler analyzes data flow and only re-runs computations when inputs actually change — equivalent to manually adding `useMemo`/`useCallback` everywhere it would help.

## Quick Performance Checklist

- [ ] Profiled with React DevTools before optimizing
- [ ] Large lists use virtualization (react-virtual, react-window)
- [ ] Routes code-split with `React.lazy` or framework routing
- [ ] `React.memo` on frequently-rendered pure components
- [ ] No inline function/object literals passed as props to memoized components
- [ ] Context split by update frequency
- [ ] Images optimized (`next/image` or `<img loading="lazy">`)
- [ ] Large dependencies lazy-loaded (charts, editors, PDFs)
- [ ] React Compiler enabled (if on React 19)
