Node.js security has changed dramatically in recent years. The ecosystem has gone from "just run `npm audit`" to a multi-layered defense strategy that includes a native Permissions Model, supply chain integrity verification, SBOM generation, and zero-trust container configurations. This guide covers what actually matters in 2026.

## The Node.js Permissions Model

Node.js v22+ ships a built-in Permissions Model that lets you restrict what your application can do at the runtime level — similar to Deno's permission system but for the existing npm ecosystem.

### Basic Usage

```bash
# Allow filesystem reads only from /app
node --permission --allow-fs-read=/app server.js

# Allow reads + writes to specific paths
node --permission \
  --allow-fs-read=/app,/tmp \
  --allow-fs-write=/tmp \
  server.js

# Allow specific environment variables
node --permission \
  --allow-env=NODE_ENV,PORT,DATABASE_URL \
  server.js

# Allow network access to specific hosts
node --permission \
  --allow-net=api.stripe.com:443,db.internal:5432 \
  server.js

# Combine permissions
node --permission \
  --allow-fs-read=/app \
  --allow-fs-write=/tmp \
  --allow-env=NODE_ENV,PORT \
  --allow-net \
  server.js
```

### Available Permission Flags

| Flag | Description |
|---|---|
| `--allow-fs-read=<path>` | Filesystem reads for path/glob |
| `--allow-fs-write=<path>` | Filesystem writes for path/glob |
| `--allow-net[=<host>]` | Network access, optionally scoped to hosts |
| `--allow-env[=<vars>]` | Environment variable access |
| `--allow-child-process` | `child_process.spawn()` and friends |
| `--allow-worker` | `Worker` threads |
| `--allow-wasi` | WASI system calls |
| `--allow-inspector` | Node.js Inspector protocol |

### Checking Permissions Programmatically

```javascript
import { permission } from 'node:process';

// Check before attempting
if (permission.has('fs.write', '/tmp/output.log')) {
  fs.writeFileSync('/tmp/output.log', data);
} else {
  console.warn('No write permission for /tmp/output.log');
}

// Check env access
const canReadDBUrl = permission.has('env', 'DATABASE_URL');
```

### Why the Permissions Model Matters

Consider a dependency that gets compromised and tries to exfiltrate environment variables:

```javascript
// Malicious code in a dependency
process.env.AWS_SECRET_ACCESS_KEY; // Normally accessible
// But with --allow-env=NODE_ENV,PORT only, this throws:
// Error [ERR_ACCESS_DENIED]: Access to env['AWS_SECRET_ACCESS_KEY'] blocked
```

The Permissions Model doesn't prevent all attacks, but it significantly raises the bar for supply chain compromise.

## Supply Chain Security

The npm ecosystem has over 2 million packages. Many widely-used packages have been compromised, injected with malware, or typosquatted. Supply chain security is now a first-class concern.

### npm audit

The baseline — run it, but understand its limits:

```bash
npm audit              # Check for known vulnerabilities
npm audit --fix        # Auto-fix safe updates
npm audit --omit=dev   # Production deps only
npm audit --json       # Machine-readable output for CI
```

`npm audit` only catches **known** CVEs. It won't catch:
- Malicious code added to a package that isn't yet a CVE
- Packages that do exactly what their README says but are malicious by design
- Typosquatted packages

### Socket.dev

[Socket](https://socket.dev) analyzes packages for suspicious behaviors before you install:

```bash
npm install @socket.dev/cli -g
socket npm install lodash  # Scans before installing
```

Socket flags:
- Packages that access the filesystem unexpectedly
- Packages that make network calls during install
- Packages with obfuscated code
- Typosquatting attempts
- Install scripts that run shell commands

Integrate into CI:

```yaml
# .github/workflows/security.yml
- name: Socket Security Check
  uses: SocketDev/socket-security-action@v1
  with:
    api-key: ${{ secrets.SOCKET_API_KEY }}
```

### Snyk

Snyk provides deeper vulnerability scanning with fix guidance:

```bash
npm install -g snyk
snyk auth
snyk test          # Scan for vulnerabilities
snyk monitor       # Continuous monitoring
snyk fix           # Interactive fix wizard
```

### Dependency Pinning

```json
// package.json — use exact versions in production
{
  "dependencies": {
    "express": "4.21.2",        // exact — NOT "^4.21.2"
    "helmet": "8.0.0",           // exact
    "prisma": "5.22.0"           // exact
  }
}
```

```bash
# npm lock file integrity
npm ci  # Install exactly what's in package-lock.json (never updates it)
# vs
npm install  # May update within semver range
```

**Use `npm ci` in CI/CD and Docker builds, never `npm install`.**

### Package Lock Integrity

```bash
# Verify lock file hasn't been tampered with
npm audit signatures  # Verifies registry signatures (npm 8+)

# Or use lockfile-lint
npx lockfile-lint --path package-lock.json --validate-https --validate-checksum
```

## Secrets Management

Never hardcode secrets. In 2026, you have several production-ready options.

### Environment Variables with Validation

At minimum, validate that required secrets exist at startup:

```typescript
// src/config.ts
import { z } from 'zod';

const envSchema = z.object({
  DATABASE_URL: z.string().url(),
  JWT_SECRET: z.string().min(32),
  STRIPE_SECRET_KEY: z.string().startsWith('sk_'),
  NODE_ENV: z.enum(['development', 'staging', 'production']),
});

// Throws at startup if any secret is missing or malformed
export const env = envSchema.parse(process.env);
```

### Infisical (Self-hostable)

```bash
npm install @infisical/sdk
```

```typescript
import { InfisicalClient } from '@infisical/sdk';

const client = new InfisicalClient({
  clientId: process.env.INFISICAL_CLIENT_ID!,
  clientSecret: process.env.INFISICAL_CLIENT_SECRET!,
});

const { secretValue } = await client.getSecret({
  environment: 'production',
```

## Container Security

### Run as Non-Root

```dockerfile
# Bad — runs as root by default
FROM node:22-alpine
WORKDIR /app
COPY . .
RUN npm ci
CMD ["node", "server.js"]

# Good — run as node user
FROM node:22-alpine
WORKDIR /app
COPY --chown=node:node package*.json ./
RUN npm ci --omit=dev
COPY --chown=node:node . .
USER node
CMD ["node", "server.js"]
```

### Read-Only Filesystem

```yaml
# docker-compose.yml
services:
  api:
    image: my-api:latest
    read_only: true
    tmpfs:
      - /tmp          # writable temp dir
      - /var/run      # for pid files
    volumes:
      - ./logs:/app/logs  # explicit writable mount
```

### Minimal Base Image

```dockerfile
# Distroless — no shell, no package manager
FROM node:22-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev
COPY . .

FROM gcr.io/distroless/nodejs22
WORKDIR /app
COPY --from=builder /app /app
CMD ["server.js"]
```

### Drop Linux Capabilities

```yaml
# kubernetes deployment
securityContext:
  allowPrivilegeEscalation: false
  readOnlyRootFilesystem: true
  runAsNonRoot: true
  runAsUser: 1000
  capabilities:
    drop:
      - ALL
```

## HTTP Security Headers with Helmet.js

```bash
npm install helmet
```

```typescript
import express from 'express';
import helmet from 'helmet';

const app = express();

// Apply all secure headers
app.use(helmet());

// Or fine-tune
app.use(helmet({
  contentSecurityPolicy: {
    directives: {
      defaultSrc: ["'self'"],
      scriptSrc: ["'self'", "'nonce-${nonce}'"],
      styleSrc: ["'self'", 'https://fonts.googleapis.com'],
      imgSrc: ["'self'", 'data:', 'https:'],
      connectSrc: ["'self'", 'https://api.stripe.com'],
      upgradeInsecureRequests: [],
    },
  },
  referrerPolicy: { policy: 'strict-origin-when-cross-origin' },
  hsts: {
    maxAge: 31536000, // 1 year
    includeSubDomains: true,
    preload: true,
  },
}));
```

## Rate Limiting, CORS, and Input Validation

### Rate Limiting

```bash
npm install express-rate-limit
```

```typescript
import rateLimit from 'express-rate-limit';

const apiLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 100,                   // limit each IP to 100 requests per window
  standardHeaders: true,      // Return rate limit info in RateLimit-* headers
  legacyHeaders: false,
  handler: (req, res) => {
    res.status(429).json({ error: 'Too many requests, please try again later.' });
  },
});

const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 5, // Very strict for auth endpoints
  skipSuccessfulRequests: true,
});

app.use('/api/', apiLimiter);
app.use('/api/auth/', authLimiter);
```

### CORS

```typescript
import cors from 'cors';

app.use(cors({
```

### Input Validation with Zod

```typescript
import { z } from 'zod';

const CreateUserSchema = z.object({
  name: z.string().min(1).max(100).trim(),
  email: z.string().email(),
  password: z.string().min(12).regex(
    /^(?=.*[a-z])(?=.*[A-Z])(?=.*\d)/,
    'Must contain uppercase, lowercase, and number'
  ),
  role: z.enum(['user', 'admin']).default('user'),
});

app.post('/api/users', async (req, res) => {
  const result = CreateUserSchema.safeParse(req.body);

  if (!result.success) {
    return res.status(400).json({
      error: 'Validation failed',
      details: result.error.flatten(),
    });
  }

  // result.data is fully typed and sanitized
  const user = await createUser(result.data);
  res.json(user);
});
```

## Detecting Prototype Pollution

Prototype pollution is when attacker-controlled input modifies `Object.prototype`, potentially affecting all objects:

```javascript
// Vulnerable pattern
function merge(target, source) {
  for (const key of Object.keys(source)) {
    target[key] = source[key]; // ❌ '__proto__' can be a key!
  }
}

// Attacker sends: {"__proto__": {"isAdmin": true}}
merge({}, JSON.parse(userInput));
// Now ({}).isAdmin === true for all objects!
```

**Prevention:**

```typescript
// Use Object.create(null) for untrusted keys
const safe = Object.create(null);

// Validate JSON keys
function safeMerge(target: object, source: unknown): void {
  if (typeof source !== 'object' || source === null) return;

  for (const [key, value] of Object.entries(source)) {
    // Block prototype pollution keys
    if (key === '__proto__' || key === 'constructor' || key === 'prototype') {
      throw new Error(`Suspicious key: ${key}`);
    }
    (target as any)[key] = value;
  }
}

// Or use a library with built-in protection
import deepmerge from 'deepmerge'; // Safe merge library
```

## Securely Handling Environment Variables in CI/CD

```yaml
# GitHub Actions — use secrets, not env vars for sensitive values
name: Deploy
on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build and deploy
        env:
          DATABASE_URL: ${{ secrets.DATABASE_URL }}   # Secret
          NODE_ENV: production                         # Not a secret
        run: |
          npm ci
          npm run build
          npm run deploy

      # Never do this:
      # - run: echo $DATABASE_URL  # Leaks to logs!
      # - run: export DB_PASS=hardcoded  # Visible in Actions UI
```

## Security Checklist

- [ ] Run `npm ci` in CI/CD (not `npm install`)
- [ ] Enable `npm audit` in your pipeline
- [ ] Use Socket.dev or Snyk for supply chain monitoring
- [ ] Pin exact dependency versions in production
- [ ] Generate and store SBOMs for compliance
- [ ] Use the Permissions Model (`--permission`) for sensitive workloads
- [ ] Apply Helmet.js HTTP security headers
- [ ] Implement rate limiting on all endpoints
- [ ] Validate all input with Zod or similar
- [ ] Run containers as non-root with read-only filesystems
- [ ] Never hardcode secrets — use a secrets manager
- [ ] Validate secrets at startup, fail fast if missing
- [ ] Drop Linux capabilities in container deployments
- [ ] Test for prototype pollution in merge/assign operations

Node.js security in 2026 requires defense in depth. No single measure is sufficient — the Permissions Model, supply chain tools, secrets management, and secure runtime configuration work together to build a genuinely hardened application.
