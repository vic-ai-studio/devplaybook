# React Performance Optimization: Complete Guide 2026

React's virtual DOM and reconciliation are fast for most applications. Performance problems emerge when components re-render too frequently, render too many elements, or perform expensive computations on each render.

This guide covers the complete toolkit for React performance optimization in 2026 — from basic memoization to the React Compiler.

## Understanding React's Rendering

Before optimizing, understand what triggers re-renders:
1. Component's state changes (`setState`)
2. Component's parent re-renders
3. Context the component consumes changes

The key insight: most re-renders are fine (React is fast). Optimize only when profiling shows a real problem.

## React Developer Tools: Profile First

```bash
# Install React DevTools browser extension
# Chrome/Firefox: "React Developer Tools"

# In code: wrap suspected-slow components
import { Profiler } from 'react';

function App() {
  return (
    <Profiler id="ProductList" onRender={(id, phase, actualDuration) => {
      if (actualDuration > 16) {  // Slower than 60fps
        console.warn(`${id} slow render: ${actualDuration.toFixed(2)}ms`);
      }
    }}>
      <ProductList />
    </Profiler>
  );
}
```

## React.memo: Skip Unnecessary Re-renders

```typescript
// Without memo: re-renders every time parent renders
function ProductCard({ product, onAddToCart }: Props) {
  return (
    <div>
      <h3>{product.name}</h3>
      <button onClick={() => onAddToCart(product.id)}>Add to Cart</button>
    </div>
  );
}

// With memo: only re-renders when props change
const ProductCard = React.memo(({ product, onAddToCart }: Props) => {
  return (
    <div>
      <h3>{product.name}</h3>
      <button onClick={() => onAddToCart(product.id)}>Add to Cart</button>
    </div>
  );
});

// Custom comparison for complex props
const ProductCard = React.memo(
  ({ product, onAddToCart }: Props) => {
    return <div>{product.name}</div>;
  },
  (prevProps, nextProps) => {
    // Return true if component should NOT re-render
    return prevProps.product.id === nextProps.product.id
        && prevProps.product.price === nextProps.product.price;
  }
);
```

**Warning**: `React.memo` only works if all props are stable references. If the parent creates new objects/functions on every render, memo won't help.

## useCallback: Stable Function References

```typescript
function ProductList() {
  const [cart, setCart] = useState<string[]>([]);

  // WRONG: new function reference every render → ProductCard re-renders
  const handleAddToCart = (productId: string) => {
    setCart(prev => [...prev, productId]);
  };

  // RIGHT: stable reference across renders
  const handleAddToCart = useCallback((productId: string) => {
    setCart(prev => [...prev, productId]);
  }, []);  // No dependencies — setCart is stable

  return (
    <div>
      {products.map(product => (
        <ProductCard
          key={product.id}
          product={product}
          onAddToCart={handleAddToCart}  // ✅ Stable reference
        />
      ))}
    </div>
  );
}
```

**Rule**: Use `useCallback` when passing functions to `React.memo`-wrapped components or as dependencies to other hooks.

## useMemo: Cache Expensive Computations

```typescript
function FilteredProductList({ products, filters }: Props) {
  // WRONG: recomputes every render (including unrelated state changes)
  const filteredAndSorted = products
    .filter(p => filters.category ? p.category === filters.category : true)
    .filter(p => p.price >= filters.minPrice && p.price <= filters.maxPrice)
    .sort((a, b) => b.rating - a.rating);

  // RIGHT: only recomputes when products or filters change
  const filteredAndSorted = useMemo(() => {
    return products
      .filter(p => filters.category ? p.category === filters.category : true)
      .filter(p => p.price >= filters.minPrice && p.price <= filters.maxPrice)
      .sort((a, b) => b.rating - a.rating);
  }, [products, filters]);  // ← recompute when these change

  return <ProductGrid products={filteredAndSorted} />;
}
```

**Rule**: Use `useMemo` for computations that take > 1ms or produce object/array references passed to memoized children.

## Virtualization: Render Only What's Visible

For large lists (> 100 items), virtualization is the highest-impact optimization:

```bash
npm install @tanstack/react-virtual
```

```typescript
import { useVirtualizer } from '@tanstack/react-virtual';

function VirtualProductList({ products }: { products: Product[] }) {
  const parentRef = useRef<HTMLDivElement>(null);

  const virtualizer = useVirtualizer({
    count: products.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 100,  // Estimated row height in pixels
    overscan: 5,              // Render 5 extra items above/below viewport
  });

  return (
    <div ref={parentRef} style={{ height: '600px', overflow: 'auto' }}>
      <div style={{ height: `${virtualizer.getTotalSize()}px`, position: 'relative' }}>
        {virtualizer.getVirtualItems().map(virtualItem => (
          <div
            key={virtualItem.key}
            style={{
              position: 'absolute',
              top: 0,
              left: 0,
              width: '100%',
              height: `${virtualItem.size}px`,
              transform: `translateY(${virtualItem.start}px)`,
            }}
          >
            <ProductCard product={products[virtualItem.index]} />
          </div>
        ))}
      </div>
    </div>
  );
}
```

Result: Render only 20–30 items in the DOM regardless of total list size.

## Code Splitting: Load Only What's Needed

```typescript
// Lazy-load heavy components
const HeavyAnalyticsDashboard = React.lazy(() =>
  import('./HeavyAnalyticsDashboard')
);

const VideoPlayer = React.lazy(() =>

```

## Context Optimization: Avoid Context as Global State

Context re-renders all consumers on every change. Common mistake:

```typescript
// BAD: every component using AuthContext re-renders when ANY auth state changes
const AuthContext = createContext({
  user: null,
  permissions: [],
  preferences: {},  // Adding preferences here triggers re-renders on preference changes
  theme: 'light',
});

// GOOD: split contexts by change frequency
const UserContext = createContext<User | null>(null);
const PermissionsContext = createContext<Permission[]>([]);
const PreferencesContext = createContext<Preferences>({});

// Components only re-render for the context they use
```

## React Compiler (React 19+)

React Compiler (formerly React Forget) automatically adds memoization. If your project uses React 19+:

```bash
npm install babel-plugin-react-compiler
```

```json
// babel.config.js
{
  "plugins": ["babel-plugin-react-compiler"]
}
```

The compiler analyzes your components and adds `useMemo`/`useCallback` automatically where beneficial. This doesn't replace understanding these hooks — it makes idiomatic React code fast by default.

## Quick Reference: When to Use What

| Tool | Use When |
|------|----------|
| `React.memo` | Component renders frequently but props rarely change |
| `useCallback` | Passing callbacks to memoized components |
| `useMemo` | Expensive computation in render, or stable reference needed |
| Virtualization | Lists > 100 items |
| `React.lazy` | Large components not needed on initial render |
| Split Context | Context value changes parts independently |
| React Compiler | React 19+ projects (handles memo automatically) |

## Profiling Checklist

1. Open React DevTools → Profiler tab
2. Record while performing the slow action
3. Look for: deep re-render trees, long flame bars, unnecessary renders
4. Fix the root cause (not the symptom)
5. Measure again — verify improvement

Performance optimization without profiling is guessing. Profile first, optimize second.
