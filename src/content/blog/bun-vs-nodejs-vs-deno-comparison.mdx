# Bun vs Node.js vs Deno: Which JavaScript Runtime to Use in 2026

The JavaScript runtime landscape has fundamentally changed. For 15 years, Node.js was the only serious option for server-side JavaScript. Now Deno and Bun offer compelling alternatives — and the choice matters for performance, developer experience, and ecosystem compatibility.

---

## The Contenders

| | Node.js | Deno | Bun |
|---|---|---|---|
| First release | 2009 | 2018 | 2022 |
| Engine | V8 | V8 | JavaScriptCore |
| Built with | C++ | Rust | Zig |
| Package manager | npm/yarn/pnpm | URL imports / JSR | bun (built-in) |
| TypeScript | Via tsc/tsx | Native | Native |
| Node compat | N/A | ~95% | ~99% |

## Performance

Bun was built for speed from day one, using JavaScriptCore (Safari's engine, optimized for startup time) instead of V8:

| Benchmark | Node.js | Deno | Bun |
|---|---|---|---|
| HTTP requests/sec | ~75K | ~100K | ~160K |
| Startup time | ~50ms | ~35ms | ~5ms |
| `npm install` (200 packages) | ~6s | N/A | ~0.5s |

Bun's install speed is transformative for CI/CD pipelines. Swapping `npm install` for `bun install` often cuts install time by 10-20x.

```bash
# Benchmark HTTP servers yourself
node --version   # v22.x
deno --version   # 2.x
bun --version    # 1.x
```

## Node.js: The Incumbent

Node.js has the largest ecosystem, the best compatibility record, and the most production hardening. Most npm packages target Node.js first.

### When Node.js is the right choice:

- **Existing codebase** — migrating a working Node.js app has no upside
- **Niche packages** — some npm packages use native addons or Node-specific internals
- **Enterprise/compliance** — well-understood security model, LTS support, certifications
- **Maximum ecosystem** — 2M+ npm packages tested against Node.js

```js
// Node.js 22 — native fetch, ESM, without experimental flags
import { createServer } from 'node:http';

const server = createServer((req, res) => {
  res.end('Hello from Node.js');
});

server.listen(3000, () => console.log('Running on :3000'));
```

## Deno: The Security-First Runtime

Deno was created by Node.js author Ryan Dahl to fix Node.js's original design mistakes. Key innovations: explicit permissions, native TypeScript, URL-based imports, and built-in tooling.

### Deno's security model

```bash
# Deno denies all by default — you must grant permissions explicitly
deno run script.ts                          # no network, no file access
deno run --allow-net script.ts              # allow all network
deno run --allow-net=api.example.com script.ts  # allow only specific host
deno run --allow-read=/tmp script.ts        # allow read from /tmp only
```

This is valuable for running untrusted code or building CLIs that should have limited system access.

### Deno's built-in tooling

```bash
deno fmt        # code formatter (no Prettier needed)
deno lint       # linter (no ESLint needed)
deno test       # test runner
deno compile    # compile to a standalone binary
deno bench      # microbenchmarks
```

### TypeScript with zero config

```typescript
// deno run this.ts — no tsconfig.json, no tsc needed
interface User {
  id: number;
  name: string;
}

const users: User[] = await fetch("https://api.example.com/users")
  .then(r => r.json());

console.log(users[0].name);
```

## Bun: The All-in-One Toolkit

Bun is the newest entrant and the most aggressive about replacing the entire JavaScript toolchain. It's a runtime, package manager, bundler, and test runner in one binary.

### Drop-in Node.js replacement

```bash
# Most Node.js projects work with zero changes
bun run server.js      # replaces: node server.js
bun install            # replaces: npm install (10-20x faster)
bun test               # replaces: jest/vitest
bun build ./src/index.ts --outdir ./dist  # replaces: webpack/esbuild
```

### Bun APIs

Bun ships high-performance APIs for common server tasks:

```typescript
// Bun's HTTP server — significantly faster than Node's http module
const server = Bun.serve({
  port: 3000,
  fetch(request) {
    const url = new URL(request.url);
    if (url.pathname === "/health") {
      return new Response("OK");
    }
    return new Response("Hello from Bun!");
  },
});

console.log(`Listening on ${server.url}`);
```

```typescript
// Bun's file I/O — simpler than Node's fs
const file = Bun.file("./data.json");
const data = await file.json();

await Bun.write("./output.txt", "Hello world");
```

### Bun Shell — scripting without bash

```typescript
import { $ } from "bun";

// Cross-platform shell scripting in TypeScript
await $`ls *.ts`;
const output = await $`cat package.json`.json();
console.log(output.name);
```

## Which Runtime Should You Choose?

**Use Node.js when:**
- Maintaining an existing Node.js project
- You need maximum npm ecosystem compatibility
- Your team has deep Node.js expertise
- You need LTS support and enterprise stability

**Use Deno when:**
- Security is a primary concern (permission model)
- You're building CLIs or scripts that should have minimal system access
- You want TypeScript without any build config
- You're working with Deno KV or Deno Deploy

**Use Bun when:**
- Starting a new project and want maximum performance
- You're building high-throughput APIs (Bun's HTTP server is fastest)
- You want a simpler toolchain (one tool for install/run/test/bundle)
- Your team uses Next.js, React, or standard npm packages (Bun compatibility is excellent)
- CI pipeline speed matters — `bun install` alone is worth it

## Practical: Node.js to Bun Migration

Most Express/Fastify/Hono apps migrate to Bun in minutes:

```bash
# Install Bun
curl -fsSL https://bun.sh/install | bash

# In your Node.js project:
bun install  # reads your existing package.json
bun run dev  # replaces: npm run dev or node index.js
```

Check compatibility with `bunx --bun npx-tool` for packages that haven't been tested on Bun.
